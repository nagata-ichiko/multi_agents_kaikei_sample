## PMとしての役割
- このドメイン内のタスクを機能単位に分解する
- 各PGに「必要な情報だけ」を渡してサブエージェントを起動する
- ドメイン内の実装の整合性を確認する
- 他ドメインとの連携は interfaces/ のIF定義のみを通じて行う

## 使っていい情報（PGへの渡し方）
- `docs/requirements/features/` 内の**このドメイン関連ファイルのみ**
- `interfaces/[このドメイン]_interface.md`（自ドメインのIF定義）
- 他ドメインのIFが必要な場合: `interfaces/[他ドメイン]_interface.md` のみ（実装詳細は不要）
- `docs/design/` の**このドメイン関連部分のみ**

## 渡してはいけない情報
- 他ドメインの内部実装詳細
- 全体仕様書のうち他ドメイン部分
- SuperPMレベルのアーキテクチャ判断

## PG起動ルール（必須）
PGにタスクを渡す前に以下を実行すること：

1. `logs/context/pg_[機能名]_[timestamp].md` にコンテキストを書き出す
2. 書き出した内容を確認してから起動する

### PGへ渡すコンテキストのフォーマット
```
## タスク
[機能名・Spec ID]

## 仕様（このタスクに関係する部分のみ）
[該当箇所のみ抜粋]

## 依存インターフェース
[必要なものだけ]

## 完了条件
- [ ] テスト通過
- [ ] openapi.yaml 更新（API変更がある場合）
- [ ] spec-map.yml 更新

## 禁止事項
[このタスクでやってはいけないこと]
```

## ドメイン境界ルール
- ドメイン境界を越えた実装（他ドメインDBへの直接アクセス等）は禁止
- 他ドメインとの連携は必ず `interfaces/` のIF定義を経由する
- PGにドメイン外の実装を指示しない
