# セッション・ログ管理ルール

## セッションライフサイクル

### 切り替えの目安
- **1 PLAN実装 + PR完了 = 1セッション** を基本単位とする
- 長時間の同一セッションはコンテキスト圧縮が頻発し、応答品質が低下するため、適切なタイミングで切り替える
- PR完了時に「セッションを新しくすることをお勧めします」と提案する

### session-state.md の管理

`tasks/session-state.md` で作業状態を管理する。

**更新タイミング:**
- スキルのフェーズが1つ完了するたび
- サブエージェントの起動/完了時
- コミット完了時
- セッション終了時

**セッション終了時:**
- session-state.md を更新する（次セッションでの復帰に必要）
- 未完了タスクがあれば `tasks/todo.md` に記録する

**セッション復帰時:**
- session-state.md が存在すれば「前回の続きから再開しますか？」と提案する
- ユーザーが同意 → 未完了ステップから再開
- ユーザーが拒否 → session-state.md をクリアして新規作業開始
- 再開前に以下を検証する:
  - 作業ブランチに未コミットの変更がないか（`git status`）
  - spec-map.yml に中途半端な状態がないか
  - `logs/context/` に前回配布したコンテキストファイルがあるか
  - `tasks/todo.md` の進捗が session-state.md と一致しているか

**長時間タスクでの保存ポイント:**
- spec-all: フェーズ0完了時（glossary作成後）、各ドメインSpec生成完了時、統合レビュー完了時
- implement-spec / revise-spec: 設計書更新完了時、コード実装完了時（テスト前）、テスト完了時（コミット前）
- orchestrate: タスク分解完了時、各ドメインPM完了時、全体統合完了時

#### session-state.md のフォーマット

```markdown
# セッション状態

## 最終更新
- 日時: YYYY-MM-DD HH:MM
- 実行中スキル: [スキル名]
- 現在のフェーズ: [フェーズ番号/名前]

## 完了済みステップ
- [x] ステップ1の内容

## 未完了ステップ
- [ ] ステップ3の内容

## コンテキスト
- 対象Spec: [REQ-ID]
- 作業ブランチ: [ブランチ名]
- 関連ログ: [logs/context/ファイル名]

## 中断時の注意事項
<!-- 再開時に知っておくべき情報（途中の判断、未解決の問題等） -->
```

## ワークフロー設計

### 段階的な計画と実行
- 3ステップ以上 or アーキテクチャに関わるタスクはPlanモードで開始する
- 途中で詰まったら無理に進めず立ち止まって再計画する
- 計画・進捗は `tasks/todo.md` に記録する

### サブエージェント活用
- リサーチ・並列分析はサブエージェントに委任し、メインコンテキストをクリーンに保つ
- 1サブエージェント = 1タスクで集中実行する

### 検証と完了基準
- 動作を証明できるまでタスクを完了としない（テスト実行・差分確認で正しさを示す）
- 完了前に自問する：テストは通るか、既存コードと一貫性があるか、意図しない副作用はないか
- バグやCIの失敗も自発的に原因特定→修正まで行う

### 自己改善ループ
- ユーザーから修正を受けたら `tasks/lessons.md` にパターンを記録する
- セッション開始時にlessonsをレビューし、同じミスを繰り返さない

### エレガントさの追求（バランスよく）
- 重要な変更前に「もっとシンプルな方法は？」と一度立ち止まる
- 明白な修正にはこのプロセスを適用しない

## プロジェクトヘルスチェック
- セッション開始時や実装前に [ヘルスチェック](../skills/_shared/health-check.md) を実施し、プロジェクトの健全性を把握する
- Spec充足率、Spec-Code一致率、テストカバー率、循環依存、設計書鮮度の5指標で判定
- 「要対応」の指標がある場合は実装前に対処を提案する

## エラーリカバリー
- 問題発生時は [エラーリカバリーパターン集](../skills/_shared/error-recovery.md) を参照して対処する
- テスト失敗、CI失敗、Spec⇔コード不整合、サブエージェント問題、マージコンフリクト等のパターンを網羅

## ログ管理ルール

`logs/context/` 配下のファイルは削除しない。理由：
- Spec化・レビュー・デバッグ時の参照資料として使用するため
- タスク分解の経緯とコンテキスト配布の記録を保持するため

**保持ポリシー:**
- 直近3ヶ月以内のファイル: 常に保持
- 3ヶ月超のファイル: 該当Specが完了済み（ステータス「完了」）であれば削除可。判断に迷う場合は保持
- `.gitignore` には追加しない（履歴をGitで追跡するため）
