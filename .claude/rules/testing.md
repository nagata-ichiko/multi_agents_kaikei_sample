# テスト戦略

プロジェクトの品質ゲートとして、以下のテスト種別を段階的に導入する：

| テスト種別 | ツール例 | 導入タイミング | 必須/推奨 |
|-----------|---------|-------------|----------|
| ユニットテスト | vitest / jest / pytest | 初回実装時 | 必須 |
| E2E テスト | Playwright / Cypress | UI実装時 | 必須（UIあり） |
| API コントラクトテスト | openapi-validator / dredd | API実装後 | 推奨 |
| アクセシビリティテスト | axe-core / pa11y | UI実装後 | 推奨 |
| 負荷テスト | k6 / artillery | 非機能要件にSLO記載がある場合 | 条件付き必須 |
| セキュリティスキャン | npm audit / trivy | CI常時 | 推奨 |

- 非機能要件に数値目標（例: 「1000req/s」「応答200ms以内」）がある場合、その検証手段（負荷テスト設定等）も実装に含めること
- `gen-tests` スキルはユニット + E2E を生成する。他のテスト種別は手動またはCI設定で追加する

# テストケース設計の原則

- テストケースは**設計書（仕様書・受入条件）を軸に**洗い出し、実装コードの分岐・エッジケースで補完する
- テスト期待値が不一致の場合、**すぐにテストを修正しない**。設計・実装・テストの三者を突き合わせ、障害かテストの誤りかを判断する

# E2Eデグレチェック（実装作業完了時）

フロントエンド/バックエンドの実装作業が完了したら、**コミット前にフロントE2Eテスト全件を実行**してデグレッションがないことを確認する。

## フロー

```
実装完了 → フロントE2E全件実行 → 全パス → コミット＆プッシュ
                                → 失敗 → 原因分析＆修正 → 再実行 → コミット＆プッシュ
```

## 実行条件
- フロントエンド or バックエンドのソースコード変更を伴う作業が完了した時
- ドキュメントのみの変更（docs/リポ内のみ）の場合は不要

## 実行方法
- CLAUDE.md の「E2Eデグレチェック」セクションに記載されたコマンドを使う
- Docker環境が起動している前提で実行する
- テスト結果を確認し、失敗がある場合は原因を分析して修正する
- **E2E全件パスするまでコミットしない**

## 判定基準
- 全テストがパス → デグレなし、コミット可
- 失敗テストあり → 今回の変更が原因か、既存の不安定テストかを切り分ける
  - 今回の変更が原因 → 修正してから再実行
  - 既存の不安定テスト（flaky） → ユーザーに報告して判断を仰ぐ

# テスト素通り（Silent PASS）の禁止

テストファイルを編集・新規作成する際、**データ不在や条件未達で早期 return して素通り PASS するコードを書いてはならない**。

## 禁止パターン

```typescript
// NG: データがなければ何も検証せず PASS
if (rows.length === 0) {
  test.info().annotations.push({ type: 'note', description: 'データなし' });
  return;
}
```

## 正しいパターン

```typescript
// OK: データがなければ FAIL — テスト環境の問題を明示する
expect(rows.length, '前提データが存在しません。テスト実行順またはセットアップを確認してください').toBeGreaterThan(0);
```

## 原則

- **データ不在はテスト環境の問題**であり、テスト自体が隠蔽してはいけない
- `test.skip` / `describe.skip` / `if (!data) return` はすべて同種の問題を引き起こす
- 前提データが必要なら、テストのセットアップで作成するか、`expect` で明示的に FAIL させる
- **ユーザーが明示的に承認した場合のみ**、例外的にスキップ・早期 return を許可する

# テストタイムアウト変更ポリシー

テストのタイムアウト値の変更は**最後の手段**であり、必ずユーザーの承認を得てから行うこと。

## 禁止事項
- タイムアウトを延長してテストを通すこと（根本原因を隠蔽する）
- ユーザーに確認せずにタイムアウト値を変更すること
- テストをスキップ（`.skip`）してタイムアウト問題を回避すること

## タイムアウトで失敗した場合の対応手順
1. **根本原因を調査** — タイムアウトは「処理が遅い」「データが不正」「ロジックバグ」のサイン
2. **原因を特定して修正** — テストデータの見直し、処理の最適化、モック化など
3. **それでも解決しない場合のみ** — ユーザーに「タイムアウト値を変更してよいか」確認する
4. **変更時は理由をコメントに残す** — `}, 60000); // 外部API応答待ちのため延長（#issue番号）`

## 極端に長いタイムアウトの目安
- 個別テストケース: 30秒超は要調査
- beforeAll/afterAll: 60秒超は要調査
- 数分以上: ほぼ確実にロジックまたはデータの問題

# E2Eフレームワーク検出ルール

- デフォルトは Playwright を前提とするが、プロジェクトに応じて自動検出する
- `cypress.config.*` が存在する場合 → Cypress を使用
- `playwright.config.*` が存在する場合 → Playwright を使用
- どちらも存在しない場合 → CLAUDE.md の「テスト設定」を確認し、未記入なら Playwright をデフォルトとしてセットアップを提案する
