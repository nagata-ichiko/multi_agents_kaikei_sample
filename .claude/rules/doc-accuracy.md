# ドキュメント生成の正確性ルール

- **実装に書いていないことは書かない。** コードに存在しない機能をドキュメントに記載してはならない
- **コード参照なしの推測禁止。** grepの結果やメソッド名だけで機能を推測してはならない。サービスファイルは全行読んで確認すること
- 一般的なシステムの機能を推測して追加してはならない（「物流システムだから分析機能があるはず」等）
- Taskエージェントをドキュメント生成に使う場合、情報収集（ファイル一覧、メソッドシグネチャ抽出）に限定し、本文は必ず自分でコードを読んで書くこと
- 生成した各機能には対応するコードの場所（ファイル名:行番号 or メソッド名）を明記すること。コード参照がない機能は記載してはならない
- 「設計中」「実装予定」等のステータスで架空の仕様書を作成してはならない

**よくある誤り:**
- ファイル生成サービス（Excel/CSV出力）を「包括的分析ダッシュボード」と誤認
- 98行のSMTP送信サービスを「Slack/SMS/Push/WebSocket通知システム」と拡大解釈

# ドキュメントサイトのスケーラビリティ

Spec数が増えた場合の mkdocs.yml nav 管理ルール：

- **20 Spec 以下:** フラットに並べてよい（`requirements/features/xxx.md`）
- **20 Spec 超:** ドメイン別にカテゴリ化する（spec-all が自動で階層化する）
- **draft-spec / spec-feature で個別追加する場合も**、既にドメインカテゴリが存在すればその中に配置する。フラットに追加してドメイン階層を崩さないこと
- mkdocs.yml の nav が長くなりすぎた場合、`navigation.sections` + `navigation.expand` の Material テーマ機能で折りたたみ表示される
