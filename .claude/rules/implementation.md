# 実装ルール

## タスク複雑度に応じたルール適用（ティア制）

このルールは `src/**`, `app/**` 等のソースコード操作時に自動ロードされる。
変更の規模に応じて、以下のティアを判断し**該当ティアのルールのみ**適用すること。

| ティア | 判断基準 | 適用セクション |
|--------|---------|---------------|
| **Tier 1（小修正）** | バグ修正、文言変更、CSS調整、1-2ファイルの変更 | 必須事項 + コード探索ルール（Step 2のみ） + 自動化ルール |
| **Tier 2（機能実装）** | 新機能追加、既存機能の変更、3ファイル以上の変更 | 全セクション適用 |
| **Tier 3（リファクタ・横断変更）** | 共通モジュール変更、アーキテクチャ変更、複数機能に影響 | 全セクション + 依存グラフ確認 + shared-components 更新確認 |

**ティア判定のヒント:** implement-spec / revise-spec / orchestrate 経由の場合は自動的にTier 2以上。直接のコード修正依頼はTier 1から判断する。

## 実装時の必須事項（全ティア共通）
- 同画面・同種処理の既存パターン（命名・配置・変換処理等）を把握し、一貫性を保って実装すること
- 実装・テストファイルは spec-map.yml に登録すること
- spec-map.yml の confirmed が spec_version と一致していることを確認すること

## デザイン方針（Tier 2以上）
- docs/designs/ にデザインファイルがある場合：それに忠実に従う
- デザイン指定がない場合：overview.mdの「対象ユーザー」に刺さる、おしゃれでモダンなデザインにする
- 管理画面：デザイン性より使いやすさ重視。シンプルで迷わないUIにする
- 特に指定がなければ全てレスポンシブデザインにする。スマホでも快適に操作できること

## コード探索ルール（2段階探索）

[2段階探索ガイド](.claude/skills/_shared/code-search-2stage.md) に従うこと。以下はティア別の適用範囲：

- **Tier 1:** Step 2（キーワード検索）のみで十分
- **Tier 2:** Step 1（構造ベース）→ Step 2（キーワード）→ Step 3（非同期・イベント）の順
- **Tier 3:** 上記 + 認証ミドルウェア、バリデーションチェーン、feature flagの影響も追加確認

修正後は変更前の値・名称でもgrepし、旧記述が他ファイルに残っていないか確認すること。
画面遷移図（`docs/design/screen-flow.md`）がある場合、同一機能の全画面パス（サブ画面含む）も洗い出すこと。

## 自動化ルール
- 実装後は必ずテストを実行して全てパスさせること
- テストが通らない場合は3回まで修正を試みること
- 3回修正してもダメな場合は、失敗内容をコミットメッセージに記載して人間に判断を仰ぐ

## auto-review への設計意図伝達
- auto-reviewの指摘に対して設計意図や反論がある場合、`DESIGN_INTENT.md` の区切り線以降に記載する
- auto-reviewはレビュー前にこのファイルを読み、意図を踏まえて判断する
- PRマージ後に自動クリアされるため、PR単位で書き捨ててよい

## テンプレートフィードバックルール
以下のパターンに該当する問題を解決した場合、ユーザーに「テンプレートリポにフィードバックしますか？」と提案すること：
- 非推奨APIを踏んで修正した → CLAUDE.md「技術制約」に追記
- CIが動かず設定を修正した → ci-templates/ を修正
- スキルの手順不足で想定外の動きをした → SKILL.mdを修正
- 環境依存の問題で他プロジェクトでも再現しうるもの
提案後、ユーザーが同意したら feedback-template スキルを実行する。

## shared-components.md の管理

`docs/design/shared-components.md` は共通コンポーネントの Single Source of Truth。テンプレートは `docs/templates/phase2/shared-components.md`。

- **作成タイミング:** orchestrate の共通基盤タスク完了時に初回作成。または implement-spec で共通コンポーネントを新規作成した場合に作成する
- **既存プロジェクト導入時:** init-spec では作成しない。最初の implement-spec 実行時に、共通コンポーネントが見つかれば作成する
- **更新タイミング:** 共通コンポーネントを追加・変更・削除した場合に、同じコミットで更新する
- **参照タイミング:**
  - **draft-spec:** 要件定義時に既存の共通コンポーネントを確認し、活用可能なものを備考に記載する
  - **feature-design.md 作成時:** セクション3「共通コンポーネント」で利用する既存コンポーネントと新規共通化の提案を記載する
  - **spec-feature:** コード分析時に共通コンポーネントを洗い出し、shared-components.md を生成/更新する
  - **spec-all:** フェーズ4.5で全ドメインを横断して共通コンポーネントを集約し、shared-components.md を一括生成する
  - **implement-spec / revise-spec の coder:** 実装開始前に必ず読み、既存の共通コンポーネントを活用する（再発明しない）
  - **implement-spec / revise-spec の reviewer:** 共通コンポーネントで代替できる箇所に独自実装がないか検証する
- **共通化の判断基準:** 2箇所以上で同じ処理・UIパターンが使われている場合は共通化を検討する。詳細は shared-components.md テンプレートのセクション7を参照

## ホットフィックス

緊急修正の詳細手順は `hotfix` スキル（`.claude/skills/hotfix/SKILL.md`）を参照。

- **許可条件:** 本番障害、セキュリティ脆弱性の即時修正、ユーザーが「緊急」と明示した場合のみ
- **ブランチ:** `hotfix/[Spec ID]`
- **Spec同期義務:** 24時間以内
