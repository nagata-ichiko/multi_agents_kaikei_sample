# リリース管理ルール

リリース時に以下を実施する（手動、またはユーザーの指示に基づいてAIが実行）：

1. **リリースタグ:** `git tag vX.Y.Z` でバージョンをタグ付けする
   - メジャー: 破壊的変更（API非互換、DB マイグレーション必須等）
   - マイナー: 新機能追加
   - パッチ: バグ修正、ホットフィックス
2. **リリースノート:** タグに含まれるコミットから変更内容を生成する
   - 含まれる REQ-ID の一覧
   - 破壊的変更の有無
   - マイグレーション手順（必要な場合）
3. **spec-map.yml のスナップショット:** リリースタグ時点の Git 状態が正となる。`confirmed == spec_version` でないエントリはリリースに含めない

# マルチ開発者での競合回避

複数人が同時に作業する場合の指針：

- **REQ-ID 採番:** overview.md の機能一覧テーブルに REQ-ID を追記する際は、必ず最新の main を pull してから採番する。採番後すぐにコミット・プッシュして他者との衝突を防ぐ
- **overview.md / mkdocs.yml:** 追記のみ（既存行の変更は避ける）にすることで、Git の自動マージで解決できるようにする
- **spec-map.yml:** REQ-ID ごとにブロックが分かれているため、異なる REQ-ID を担当していれば自動マージ可能。同じ REQ-ID の同時変更は避ける
- **ブランチ運用:** 1人1ブランチ（`feature/REQ-XXX-NNN`）を厳守。同じ REQ-ID を複数人が同時に触らない

# リファクタリングルール

API や仕様を変更しない内部改善（リファクタリング、パフォーマンス改善、技術的負債の解消）のルール：

- **ブランチ:** `refactor/[対象の簡潔な説明]`（例: `refactor/optimize-user-query`）
- **Spec更新:** 外部仕様が変わらないためSpec更新は不要。ただし非機能要件（パフォーマンス等）に影響がある場合は OVERVIEW Spec を更新する
- **spec-map.yml:** 実装ファイルのパスが変わった場合は `path` を更新する（`confirmed` はそのまま）
- **テスト:** 既存テストが全てパスすることを確認する。振る舞いが変わらないことの証明
- **コミットメッセージ:** `refactor: [変更概要]`
- **設計書への影響:** architecture.md の技術構成が変わる場合のみ更新する

# CI テンプレートの技術スタック対応

ci-templates/ は Node.js + npm をデフォルトとするが、init-spec がプロジェクトの技術スタックに合わせて書き換える：

- `package.json` がある → Node.js（デフォルト。書き換え不要）
- `pyproject.toml` / `requirements.txt` がある → Python（pip install、pytest等に書き換え）
- `Gemfile` がある → Ruby（bundle install、rspec等に書き換え）
- `go.mod` がある → Go（go test等に書き換え）

init-spec はコードリポを分析して適切なビルド・テストコマンドを CI テンプレートに反映すること。
