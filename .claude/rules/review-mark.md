# レビューマーカー自動挿入ルール

対象パスの Markdown を Edit / Write する際、**変更・追加した段落を HTML コメントマーカーで囲む**こと。ユーザーは MkDocs 画面上の赤背景と「🔴 未承認レビュー」一覧ページで変更箇所をレビューする。

## 対象パス

- `docs/requirements/**/*.md`（要件定義書）
- `docs/design/**/*.md`（基本設計・詳細設計その他）
- `docs/design-detail/**/*.md`（機能別詳細設計）

## マーカー形式（3 種）

変更の性質に応じて 3 種のマーカーを使い分ける。

### 1. 追加（新規作成された段落）

```markdown
<!-- review:pending id=r-YYYYMMDD-NNN -->

新しく追加する段落の内容

<!-- /review -->
```

### 2. 変更（既存段落を書き換える）

旧内容と新内容を **was / now** で挟んで両方記録する:

```markdown
<!-- review:pending id=r-YYYYMMDD-NNN -->
<!-- review:was -->

旧段落の内容

<!-- review:now -->

新段落の内容

<!-- /review -->
```

MkDocs 画面では旧内容が「変更前を見る」折りたたみ表示になり、新内容が本文として表示される。承認時は was ブロックが消え、now の本文だけが残る。

### 3. 削除（段落を消す）

`type=delete` を付け、マーカー内に削除対象の段落を入れる:

```markdown
<!-- review:pending id=r-YYYYMMDD-NNN type=delete -->

削除する段落の内容

<!-- /review -->
```

MkDocs 画面では打ち消し線＋「🗑 削除予定」ラベル付きで表示される。承認時は**マーカーごと本文も削除**される。

## ID 採番

- フォーマット: `r-YYYYMMDD-NNN`
  - `YYYYMMDD`: 当日日付（例: `20260413`）
  - `NNN`: 当日の連番（ゼロパディング3桁）
- **連番は全ファイル横断で決める**
  - `grep -rho 'review:pending id=r-YYYYMMDD-[0-9]\+' docs/` で当日の既存 ID を取得し、最大値+1
  - 例: 当日既存が `r-20260413-003` まで → 新規は `r-20260413-004`

## 付与対象

- **追加**: 新規に追加した段落
- **変更**: 既存段落を本質的に書き換えた場合（仕様変更・文言の大きな入れ替えなど）
- **削除**: 段落を削除する場合（この時点ではまだファイル上には残しておき、type=delete マーカーで囲む）
- **ファイル全体が新規作成の場合**: ファイル先頭（フロントマターや H1 の直後）に**追加マーカー1個でファイル全体を包む**。段落ごとに個別のマーカーを付ける必要はない（ファイル内に既存段落はなく全てが新規のため）。承認後はマーカーだけが除去され、本文はそのまま残る

## 付与しなくてよいもの

- 誤字修正・記号だけの微修正
- マーカー自体の挿入・削除
- 段落の位置移動のみ（意味が変わらない並び替え）

## 変更 or 追加の判定

- 既存段落の**趣旨を維持しつつ文言を差し替え**る場合 → **変更（was/now）**
- 既存段落を**削除して別概念を新規追加**した場合 → **削除 + 追加**（2つのマーカー）
- **新規セクション**を設けた場合 → **追加**

## 除外ブロック

次の領域内はマーカーで囲まない（構文が壊れる or 不要）:

- コードブロック（\`\`\` ... \`\`\`）内
- テーブルセル単位の変更（表を丸ごと囲むのは可）
- 既に他のマーカー内にある領域（マーカーのネストは禁止）
- YAML フロントマター（`---`〜`---`）

## 粒度

- **段落単位**（Markdown の空行で区切られるブロック）
- リスト全体・段落＋サブリストを1つのマーカーで囲んでよい
- 長文セクションはセクション単位で1マーカーにしてもよい

## 既存マーカーの扱い

- 既にマーカー内にある段落を再変更 → マーカーは維持、新 ID を振らない
- 承認済み（マーカーなし）段落を再変更 → 新規マーカーを付与
- マーカー位置を移動する必要があっても、元の `id` は引き継ぐ

## 承認フロー

ユーザーは次のいずれかでマーカーを除去する:

1. MkDocs 画面の「✓ 承認」ボタン（`scripts/docs_serve.py` 起動時）
2. `/review-mark approve <id>` コマンド
3. **PR を main にマージ → GitHub Actions `auto-approve-on-merge.yml` が自動除去**

## 実装時の注意

- マーカー挿入・除去で段落本文を消さない
- 挿入前後のファイルで本文が完全一致すること（マーカー以外は無変更）
- 一度に複数ファイルを編集する場合は、各ファイルで同じ日付の連番を使う
