## Claude Code 設定

### ルール優先順位（複数ルールが同時発火した場合）

1. **security.md** — セキュリティは常に最優先。他のルールと矛盾する場合はセキュリティが勝つ
2. **spec-management.md** — Spec 体系・品質ゲートはプロセスの骨格
3. **implementation.md / testing.md** — コード・テストの実装ルール
4. **その他のルール** — 状況に応じて適用

### セキュリティ設定

詳細は `.claude/rules/security.md` を参照（機密ファイル操作時に自動ロード）。
概要: サンドボックス有効、deny ルールで破壊的コマンドをブロック、PreToolUse フックで本番接続を検出。

### モデル設定

Claude Code の `/model` コマンドで以下を推奨設定にすること：

- **プランモード（設計・判断）**: Opus
- **実行モード（コード生成）**: Sonnet

### テンプレート設計原則

`docs/templates/` 改修時は厳守：

- **1 テンプレート = 1 関心事 = 1 サブエージェント担当範囲**
- **フェーズ境界を守る** — phase1（WHAT）と phase2（HOW）を混ぜない

## AI 主動開発ルール

### 行動原則
- **AIが主導、人間は判断**
- **設計書ファースト** — コード変更前に設計書を更新。実装後は設計書⇔コードの一致を検証し、不一致は同コミットで修正
- **コードを読まずに書かない** — 必ず既存コードを確認してから変更する
- **動作を証明できるまで完了とマークしない** — テスト実行・動作確認を必ず行う
- **シンプル第一**：変更・影響範囲を最小に
- **手を抜かない**：根本原因を特定し、シニアエンジニア水準を維持
- **影響を最小化**：必要箇所のみ変更。バグを持ち込まない
- **次のステップを提案** — 作業完了後は「次は○○しますか？」と提示

### 新規プロジェクト（0→1）の開発順序

全機能の要件定義 → 基本設計 → 人間の確認 → 実装。途中で「実装して」と言われても、要件定義・基本設計の確定を先に提案すること。

### 実装プラン（impl-plan）

コード変更を伴う作業では、着手前に実装プランを作成する。PlanMode で作成する。

- 格納場所: `impl-plans/[識別子].md`（プロジェクトルート直下。docs/ には入れない）
- 命名: `REQ-AUTH-001.md`（Spec 起点）/ `fix-201.md`（Issue 起点）/ `[作業名].md`（その他）
- 作業完了後は削除してよい。永続的なドキュメントではなく、作業中の一時ファイル
- 内容:
  - 実装順序（ファイル単位、依存関係順）
  - 流用すべき既存コード
  - 判断メモ（WHY NOT）: やらないこと・スコープ外の理由
  - テスト計画

実装プランは以下の目的で使う:

- **セッション引き継ぎ**: コンテキスト圧迫時に新セッションで再開するための永続的な作業状態
- **Agent 委任**: Sonnet Agent に渡す「上から順にやるだけ」の指示書
- **レビュー時の照合**: 実装が計画通りか検証する基準

### Agent の使い方

- **実装プランが明確な作業は Agent(Sonnet) に委任する** — プランに実装順序・対象ファイル・流用コードが具体的に書かれていれば、Agentは「上から順にやるだけ」で完遂できる
- **独立した複数ステップは Agent を並列起動する** — 実装プランのステップが互いに独立していれば、複数の Agent(Sonnet) を同時に起動する
- **実装プランで解決しきれない曖昧さがある場合は本体が直接実行する** — 既存コードとの矛盾、設計書に記載のないエッジケース、テストの予期せぬ失敗など
- **レビューは常に本体が直接実行する** — 設計意図の理解が必要なため
- Agent間の直接通信はできない。結果は全て親に返る
- Agentの中でSkillは呼べない。必要な情報はファイルパスで渡し、Agent自身がReadする

### コンテキスト管理

- コンテキストの自動圧縮が発生すると、過去に読んだ設計書やコードの詳細が失われ判断品質が低下する
- 作業が長引いて判断の精度低下を感じたら、作業状態を実装プランに追記し、新セッションでの再開を提案する
- コンテキストを守るためにAgentに逃がすのではなく、セッションを分けることで判断品質を維持する

### マルチエージェント構成

実装を伴うリクエストは orchestrate（SuperPM→ドメインPM→PG の3層）が自動選択される。ドキュメント生成のみ（spec-feature / draft-spec 等）では動かない。`domains/` 未存在時は自動初期化。
