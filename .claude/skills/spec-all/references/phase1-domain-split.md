# フェーズ 1: ドメイン分割計画

### 1.1 現状把握
1. `docs/requirements/overview.md` の機能一覧を読む
2. コードリポのディレクトリ構造を確認（CLAUDE.md の技術スタックと [技術スタック対応ガイド](../../_shared/tech-stack-guide.md) のバックエンド別パス対応表から、モデル層・コントローラ層のパスを特定する）
3. ルーティング定義（例: Rails=`config/routes.rb`, Django=`*/urls.py`, Next.js=`app/`ディレクトリ構造）を読み、機能とファイルの対応を把握する

### 1.2 ドメイン分割
機能一覧をドメイン（3〜6個）に分割する。分割基準：
- **コントローラの名前空間**に合わせる（`admins/users/` → ユーザードメイン等）
- 1ドメインあたり 3〜6 機能が目安
- 関連モデルが共通する機能は同じドメインにまとめる

**大規模ドメインの分割ルール:**
- 対象ファイルの合計行数が **3,000行超** または **機能数が6超** のドメインは、サブエージェントのコンテキスト枯渇を防ぐため **最初から2つのサブドメインに分割** する
- 分割の粒度: 機能の依存関係が薄い境界で分ける（例: 「ユーザー認証・登録」と「ユーザープロフィール・検索」）
- 分割後の各サブドメインは独立したサブエージェントとして起動する
- 計画記録時に分割理由と分割境界を明記する

### 1.3 ファイル割り当て
各ドメインに対し、分析対象ファイルを**明示的に列挙**する：

**バックエンド（[技術スタック対応ガイド](../../_shared/tech-stack-guide.md) のバックエンド別パス対応表を参照）:**
- モデル層（例: Rails=`app/models/xxx.rb`, Django=`*/models.py`, Go=`*/model/*.go`）
- コントローラ層（例: Rails=`app/controllers/xxx/`, Django=`*/views.py`, Express=`*/controllers/`）
- 関連するメーラー、ジョブ、アップローダー
- ルーティング定義の該当セクション

**フロントエンド（[技術スタック対応ガイド](../../_shared/tech-stack-guide.md) のフロントエンド別パス対応表を参照）:**
- ページコンポーネント（例: Next.js App Router=`app/**/page.tsx`, Nuxt=`pages/**/*.vue`）
- 関連する UI コンポーネント（`components/`）
- 状態管理（Redux slice、Pinia store 等）
- API 通信ユーティリティ（`utils/`, `api/`, `services/` 等）

**共通（全ドメインにスキーマ定義を割り当て）:**
- スキーマ定義（例: Rails=`db/schema.rb`, Django/SQLAlchemy=models.py, Prisma=`prisma/schema.prisma`）を対象ファイルに含める
- 各エージェントは自ドメインのテーブルのみ抽出する

**ドメイン横断モデルの扱い:**
- 複数ドメインにまたがるモデル（例: contract.rb がユーザーにも案件にも関連）は **主たるドメインに割り当て、他ドメインには「共有モデル」として参照指示を出す**
- `logs/context/specall_plan_[timestamp].md` の「共有モデル一覧」セクションに記録し、各分析エージェントに通知する
- 例: `contract.rb → 案件・契約ドメインが主担当。ユーザードメインのエージェントは contract.rb のうちユーザー関連の association/scope のみ抽出`

### 1.4 ナビゲーション階層の決定

ドメイン分割をそのまま `mkdocs.yml` のナビゲーション階層に使う。
**全機能をフラットに並べない。** ドメインをカテゴリとして階層化する（例: `要件定義 > ユーザー管理 > 認証・登録`）。
カテゴリ名は日本語で、ユーザーが直感的に探せる名前にする。計画記録時にナビゲーション構成案も含める。

### 1.5 計画記録
`logs/context/specall_plan_[timestamp].md` に以下を記録：
- ドメイン一覧と各ドメインの担当機能
- 各ドメインの分析対象ファイル一覧
- **共有モデル一覧**（どのモデルがどのドメインの主担当か、他ドメインは何を参照するか）
- **ナビゲーション階層構成案**（ドメイン→機能の対応）
- バッチ実行順序（依存関係がある場合）

### 1.6 ユーザー確認
ドメイン分割計画（ドメイン名・機能数・主要ファイル）をユーザーに提示し、承認を得る。
