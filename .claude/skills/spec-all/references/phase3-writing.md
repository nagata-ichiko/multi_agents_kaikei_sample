# フェーズ 3〜4: 執筆プラン整理 → spec-writer 委任

コード分析結果をもとに、**親エージェントが執筆プランを整理し、spec-writer に委任する**。
サブエージェントに Spec 執筆を丸投げしてはならない。根拠リストの用意は親が行う。

### 各バッチ開始時に必ず読むファイル（統一性の担保）

コンテキスト圧縮で前のバッチの内容が消えても、以下のファイルを読めば統一基準を維持できる：

1. `docs/requirements/glossary.md` - 用語辞書（表記ゆれ防止）
2. `logs/context/specall_reqid_plan.md` - REQ-ID カテゴリ一覧（ID 重複防止）
3. [Spec 執筆標準](../../_shared/spec-writing-standard.md) - セクション構成・記述ルール
4. `docs/requirements/overview.md` - 最新の機能一覧・ステータス（進捗把握）

**これにより、PM の記憶ではなくファイルが統一性の源泉になる。**

### 3.1 バッチサイズの決定

コンテキスト圧縮を避けるため、バッチに分けて処理する。
**バッチサイズは固定しない。** 以下の基準で動的に判断する：

- 分析レポートのボリュームを見て、コンテキストに余裕がある限り詰める
- 大規模機能（モデル行数 500行超、コントローラアクション 10超）→ 1〜2 機能/バッチ
- 中規模機能（モデル行数 100〜500行）→ 3〜5 機能/バッチ
- 小規模機能（モデル行数 100行未満、コントローラアクション 3以下）→ 5〜10 機能/バッチ
- **トークン残量が 7万以下なら 1機能ずつ処理する**（統一基盤ファイル常駐分のバッファを考慮）

### 3.2 執筆プランの整理（親エージェント）

各バッチ内の機能ごとに以下を整理する:

1. サブエージェントのレポートから該当機能の分析結果を読む
2. **不足があれば自分でコードを追加読みする**（レポートだけで判断しない）
3. 以下を用意する:
   - REQ-ID 採番（`REQ-[機能]-[連番]`、`logs/context/specall_reqid_plan.md` を参照）
   - 受入条件（具体的な数値を明記したチェックリスト形式）
   - コード根拠リスト（全記述に対応する `ファイル名:行番号`）
   - Mermaid 図（ビジネスロジックの処理フロー・分岐）
   - 関連Spec・共通コンポーネント

### 3.3 spec-writer への委任（バッチ単位で並列）

バッチ内の各機能を `Agent(subagent_type: spec-writer)` に**並列**で委任する（ファイル単位で独立しているため）。

**委任プロンプト形式**:

```
あなたは spec-writer です。以下のドキュメントを生成してください。

## 生成対象
docs/requirements/features/[機能名].md

## 参照テンプレート
（存在する場合）docs/templates/phase1/requirements-spec.md

## 埋める情報
- REQ-ID: [採番値]
- 担当者: [名前 or 未定]
- ステータス: [値]
- 受入条件: [親が整理した条件リスト]
- 関連Spec: [特定済みREQ-IDリスト]
- 活用する共通コンポーネント: [リスト]

## コード根拠リスト（全記述の根拠。独自にgrepして補完しないこと）
[フェーズ2の分析レポートから抽出したコールチェーン・ビジネスルール・DBスキーマ等を根拠付きで列挙]

## ビジネスロジック図（Mermaid）
[フェーズ2で把握した処理フロー・分岐をMermaidフォーマットで渡す]

## 厳守事項
- 渡されたコード根拠リストのみを使う。独自にコードを grep / Read して情報を補完しない
- 根拠（ファイル名:行番号）がない記述は書かない
- 要件定義書にはWHAT（何を作るか）だけ記載。HOW（API/DB/画面遷移図）は記載しない
- 不明点は「※ 要確認」マーカーを残し、推測で埋めない
- index系ファイル（overview.md / spec-map.yml / mkdocs.yml / dependency-graph.md）は触らない（親が更新）
- 完了時に軽量セルフチェックを実行し、結果を報告すること
```

### 3.4 index系更新（親エージェント）

spec-writer は index 系に触らないため、各バッチで親が責任を持って更新する:

- `docs/requirements/overview.md` の REQ-ID・ステータスを更新
  - Spec 執筆完了 → ステータスを「完了」に変更
  - 次バッチ以降で着手予定 → 「未着手」のまま
- `spec-map.yml` にエントリ追加（[spec-map.yml 操作ガイド](../../_shared/spec-map-operations.md) 参照）
- `mkdocs.yml` の nav に機能ページを追加（**フェーズ1.4で決定した階層構成に従う**。ドメイン別カテゴリの中に配置する）
- `docs/requirements/glossary.md` に新しい用語があれば追記

### 3.5 バッチごとのコミット

**各バッチの Spec 執筆 + index 更新が終わったら必ずコミットする。**
セッション中断時に未コミットの成果物を失わないため。

```
feat: spec-all バッチN完了（REQ-XXX, REQ-YYY, REQ-ZZZ）
```

### 3.6 バッチ完了時レビュー

**各バッチ完了後、統一性レビューエージェントを起動する。**

[Spec 統一性レビュー](../../_shared/spec-consistency-review.md) に従い、
サブエージェントに全既存 Spec の横断チェックを実行させる。

レビュー結果への対応：
- **自動修正可能** → 即座に修正して次のバッチへ
- **要確認** → ユーザーに提示して判断を仰ぐ（次のバッチは判断待ちせず並行で進めてよい）
