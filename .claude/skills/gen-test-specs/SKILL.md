# システムテスト・UAT仕様書生成

## 前提条件
| 前提ファイル | 用途 |
|---|---|
| `docs/requirements/features/*.md` | テストシナリオの源泉（受入条件） |
| `docs/api/openapi.yaml` | APIテストケースの導出 |
| `docs/design/screen-flow.md` | 画面遷移テストの導出 |
| `docs/design/features/*-logic.md` | ビジネスロジックテストの導出 |
| `docs/templates/phase4/system-test-spec.md` | システムテスト仕様書テンプレート |
| `docs/templates/phase4/uat-spec.md` | UAT仕様書テンプレート |

## 手順

1. 前提ファイルの存在チェック。不足があれば報告して停止
2. テンプレートを読み込む（`docs/templates/phase4/system-test-spec.md`, `docs/templates/phase4/uat-spec.md`）
3. 要件定義書（features/*.md）を全て読み、受入条件を抽出
4. ユーザーに生成対象を確認:
   ```
   生成対象:
   ✅ システムテスト仕様書 (docs/tests/system-test-spec.md)
   ✅ 受入テスト仕様書 (docs/tests/uat-spec.md)
   この内容で進めますか？
   ```
5. システムテスト仕様書を生成（`docs/tests/system-test-spec.md`）:
   - テンプレートをコピーして各セクションを埋める
   - 機能テストシナリオ: 各Specの受入条件からE2Eシナリオを導出
   - 非機能テスト: OVERVIEW Specの非機能要件からテスト項目を導出
   - テストステップは具体的な操作手順で記載（「ログインする」ではなく「メールアドレスとパスワードを入力し、ログインボタンをクリックする」）
   → コミット: `docs: システムテスト仕様書を追加`
6. UAT仕様書を生成（`docs/tests/uat-spec.md`）:
   - テンプレートをコピーして各セクションを埋める
   - ビジネスシナリオ: 要件定義の「概要」「アクター」からユーザーストーリー形式で記載
   - テスト手順: エンドユーザーが実施する操作手順（技術用語は使わない）
   - 受入基準: 各Specの受入条件をビジネス観点で再構成
   → コミット: `docs: 受入テスト仕様書（UAT）を追加`
7. mkdocs.yml の nav にテスト仕様書を追加
8. 次のステップを提案（「テスト結果報告書のテンプレートも準備しますか？」「テストコードの生成に進みますか？」）

## ルール
- テンプレートのセクション構成を維持すること（セクションの追加は可、削除は不可）
- テストシナリオは要件定義の受入条件に基づくこと（推測でシナリオを作らない）
- UATのテスト手順は非エンジニアが実施できる表現にすること
- 実装コードが存在する場合は、画面名やボタン名を実際のUIに合わせること
