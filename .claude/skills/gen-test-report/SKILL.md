# テスト結果報告書生成

## 前提条件
| 前提ファイル | 用途 |
|---|---|
| テスト実行結果（CI/ローカル） | テスト通過率・カバレッジの集計 |
| `docs/tests/system-test-spec.md` | システムテスト結果の記入 |
| `docs/tests/uat-spec.md` | UAT結果の記入（あれば） |
| `docs/templates/phase4/test-report.md` | テスト結果報告書テンプレート |

## 手順

1. テンプレートを読み込む（`docs/templates/phase4/test-report.md`）
2. テスト実行結果を収集:
   - ユニットテスト: テスト実行コマンドを実行し、結果を取得（CLAUDE.mdの「テスト設定」「開発コマンド」を参照）
   - カバレッジ: カバレッジレポートがあれば読み取る
   - E2Eテスト: E2Eフレームワーク（CLAUDE.mdの`test_e2e`参照）の実行結果を取得
   - CIの結果: `.github/workflows/` の最新実行結果があれば参照
3. テスト結果を集計:
   - テスト種別ごとの合計/パス/失敗/スキップ数
   - カバレッジ率（行/ブランチ/関数）
   - 失敗テストの一覧と原因分析
4. ユーザーに手動テスト結果の確認:
   - システムテスト仕様書・UAT仕様書に手動テスト結果が記入済みか確認
   - 未記入の場合は「手動テスト結果は後から追記できます。自動テスト結果のみで報告書を生成しますか？」と確認
5. テスト結果報告書を生成（`docs/tests/test-report.md`）:
   - テンプレートをコピーして各セクションを埋める
   - テスト実施サマリー: 自動テストの集計結果
   - 不具合一覧: 失敗テストから不具合を一覧化
   - 品質評価: カバレッジ達成状況、不具合密度
   - 総合判定: 全テスト通過かつカバレッジ目標達成なら「リリース可」、それ以外は「条件付きリリース可」or「リリース不可」を提案
   → コミット: `docs: テスト結果報告書を追加`
6. 結果サマリーを提示:
   ```
   テスト結果報告書を生成しました:
   - ユニットテスト: XX/XX パス (カバレッジ XX%)
   - E2Eテスト: XX/XX パス
   - 総合判定: [リリース可/条件付き/不可]
   ```
7. 次のステップを提案（「不具合の修正に進みますか？」「手動テスト結果を追記しますか？」）

## ルール
- テスト結果は実際の実行結果のみ記載（推測しない）
- テスト未実行の項目は「未実施」と明記する
- 不具合の重要度は自動判定: テスト名やエラー内容から Critical/Major/Minor を分類
- 総合判定は提案であり、最終判断は人間が行うことを明記する
- 手動テスト結果が未記入でも報告書は生成可能（自動テスト結果のみで作成）
