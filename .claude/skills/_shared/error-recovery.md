# エラーリカバリーパターン集

よくある失敗パターンと対処法。問題発生時に参照する。

## 1. テスト失敗

### ユニットテストが通らない
1. エラーメッセージを読み、失敗しているテストケースを特定する
2. テストコードとプロダクションコードの両方を読む
3. 原因を切り分ける:
   - **テストが古い:** Spec変更に追従していない → テストを修正
   - **実装バグ:** ロジックが仕様と異なる → 実装を修正
   - **環境依存:** モック不足、環境変数未設定 → テスト環境を修正
4. 3回修正して通らない場合 → 失敗内容をまとめて人間に判断を仰ぐ

### E2Eテストがタイムアウト
1. dev サーバーが起動しているか確認（CLAUDE.md の開発コマンドを参照）
2. `baseURL` が正しいか確認（playwright.config.ts）
3. 認証バイパスが正しく設定されているか確認
4. ネットワーク依存のテストは `page.waitForResponse()` でAPI応答を待つ

## 2. CI失敗

### GitHub Actions が赤い
1. **どのジョブが失敗したか** をまず確認（unit / e2e / lint / build）
2. ジョブ別の対処:
   - **lint:** ローカルで lint を実行して修正
   - **build:** TypeScript型エラーが多い。`tsc --noEmit` でローカル確認
   - **unit:** ローカルで再現するか確認。CI固有の場合は環境差異を調査
   - **e2e:** CI上のブラウザバージョン差異、またはタイムアウト値の調整
3. `.github/workflows/` の設定を確認（Node.jsバージョン、サービスコンテナ等）

### 依存パッケージのインストール失敗
1. `package-lock.json` / `yarn.lock` が最新か確認
2. Node.jsバージョンが `.nvmrc` と一致しているか確認
3. プライベートレジストリの認証が設定されているか確認

## 3. Spec⇔コード不整合

### spec-map.yml で `confirmed < spec_version` が見つかった
1. 該当Specの変更履歴を確認（どのバージョンで何が変わったか）
2. 実装ファイルを読み、変更が反映されているか確認
3. 反映されていない場合:
   - 軽微な変更 → 即座に実装を修正
   - 大きな変更 → revise-spec スキルで正式に対応
4. confirmed を spec_version に更新

### 設計書に存在するがコードにない機能
1. まず本当にコードにないか2段階探索で確認（構造ベース → grep）
2. コードにない場合:
   - **未実装:** implement-spec で実装する
   - **削除済み:** 設計書から削除する（update-docs）
   - **誤記:** 設計書を修正する

## 4. サブエージェント関連

### サブエージェントが矛盾した結果を返した
1. 矛盾点を具体的に特定する（どのファイル/REQ-IDについて食い違いがあるか）
2. 該当コードを自分で読み直して正しい方を判定する
3. 矛盾が解消できない場合:
   - 両方の結果をログに残す
   - 人間に判断を仰ぐ

### サブエージェントがコンテキスト超過で中断した
1. タスクをさらに小さく分割する
2. コンテキストファイルから不要な情報を削る
3. 「ファイル一覧の取得」と「内容の分析」を別のサブエージェントに分ける

## 5. ホットフィックス後のSpec同期忘れ

### 検知方法
- コミットメッセージに `[HOTFIX]` を含むが、24時間以内に対応するSpec更新コミットがない
- `tasks/todo.md` にSpec同期タスクが残ったまま

### 対処
1. hotfix コミットの差分を読む
2. 影響するSpec（spec-map.yml で特定）を更新する
3. spec-map.yml の spec_version を+1、confirmed も合わせる
4. 「Spec同期完了」のコミットを作成する

## 6. マージコンフリクト

### spec-map.yml のコンフリクト
- 異なるREQ-IDの追加 → 両方残す（キーが異なるため自動マージ可能なはず）
- 同一REQ-IDの変更 → spec_version が大きい方を採用し、confirmed を再確認

### 設計書のコンフリクト
1. 両方の変更内容を読む
2. 片方が他方を包含している場合 → 包含している方を採用
3. 独立した変更の場合 → 両方をマージ
4. 矛盾する変更の場合 → 人間に判断を仰ぐ

### overview.md への複数人同時更新
1. 各自の変更がどのセクションに影響するか確認する
2. セクション単位で分離できる場合 → 両方のセクション変更を手動マージ
3. 同一セクション内の矛盾 → 後からコミットした方が全体を統合する責任を持つ

## 7. スキル固有の障害

### spec-all でサブエージェントが途中で中断した
1. `logs/context/specall_plan_*.md` で計画を確認する
2. `logs/context/` 内の完了済みドメインのレポートを確認する
3. 未完了ドメインだけを対象に spec-feature を個別実行する
4. 全ドメイン完了後、spec-all の統合レビューフェーズ（フェーズ5-6）を手動で実施する

### Agent Teams のレビューループが3回超で収束しない
1. レビュー指摘内容を確認し、パターンを分類する:
   - **同じ指摘の繰り返し:** coder が指摘を正しく理解していない → 指摘を具体的なコード修正として書き直す
   - **矛盾する指摘:** reviewer の基準が不安定 → 設計意図をDESIGN_INTENT.mdに記載してレビュー基準を明確化
   - **些末な指摘の連鎖:** 重要度の低い指摘が続いている → 「Critical/Majorのみ対応」とスコープを限定
2. 3回で収束しない場合は現状のコードをコミットし、残りの指摘をIssueとして記録する

### 循環依存が dependency-graph.md で検出された
1. 循環しているSpec群を特定する（A→B→C→A 等）
2. 解消パターンを選択する:
   - **Spec分割:** 共通部分を新しいSpecに切り出す（例: A→B→A を A→Common, B→Common に変更）
   - **イベント駆動化:** 同期依存をイベント経由の非同期依存に変更する
   - **インターフェース抽出:** 依存方向を反転させるためのIF定義を `interfaces/` に追加する
3. dependency-graph.md を更新し、循環が解消されたことを確認する

### スキル実行時に前提ファイルが見つからない
1. エラーメッセージから不足ファイルを特定する
2. 不足ファイルを生成する先行スキルを特定する（例: overview.md がない → init-spec を実行）
3. 先行スキルを実行してから元のスキルを再実行する
