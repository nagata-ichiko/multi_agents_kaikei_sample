# タスク分解パターン

上位層が下位層にタスクを分解・配布・起動する際の共通手順。
orchestrate（SuperPM→ドメインPM）と domain-pm（ドメインPM→PG）の両方で使う。

## 1. 分析（コードには触れない）
- 要求を確認し、影響範囲を特定する
- タスクを下位層の単位に分解する（1下位層 = 1タスク単位）
- **共通基盤タスクの抽出**: 複数タスクが共通して必要とする基盤（共通UIコンポーネント、共通ユーティリティ、認証ガード、レイアウト等）を特定し、独立したタスクとして切り出す
- 依存関係を整理し、並列実行できるものと直列が必要なものを分ける
  - **共通基盤タスクは最優先で実行する**（他の機能タスクはこれに依存する）
- **ドメイン間インターフェースの生成**: 複数ドメインにまたがるタスクの場合、`interfaces/[ドメイン名]_interface.md` を生成・更新する。IF定義は実装前に確定させ、各ドメインPMに配布する
- **依存グラフの更新**: `docs/design/dependency-graph.md` に影響するSpecの依存関係を反映する

## 2. 計画の記録
- `logs/context/` に計画ファイルを書き出す
  - 影響範囲、タスク一覧、実行順序、並列実行グループを含める

## 3. コンテキスト配布
各下位層に渡す情報を個別ファイルに書き出す。

### ファイル命名規則
| 配布元 → 配布先 | ファイル名 |
|----------------|-----------|
| SuperPM → ドメインPM | `logs/context/pm_[domain]_[timestamp].md` |
| ドメインPM → PG | `logs/context/pg_[機能名]_[timestamp].md` |
| SuperPM 計画 | `logs/context/superpm_plan_[timestamp].md` |

### コンテキストの範囲
- **含めるもの**: 配布先のタスクに直接関係する仕様、依存するIF定義
- **含めないもの**: 他の配布先の内部仕様、配布先が知る必要のない全体仕様

### コンテキストファイルの必須構造

全てのコンテキストファイル（`pm_*.md`, `pg_*.md`）は以下のセクションを含むこと。受領側はこの構造を検証し、不足があれば差し戻す。

```markdown
# [タスク名]

## タスク概要
- 対象REQ-ID: [REQ-XXX-001, REQ-XXX-002, ...]
- 変更種別: [新規実装 / 仕様変更 / バグ修正]
- 概要: [1-2文で何をするか]

## タスク依存関係
- blocks: [このタスクが完了しないと着手できないタスク]
- blockedBy: [このタスクの前に完了が必要なタスク]

## 必須ドキュメント参照
- 要件定義書: docs/requirements/features/[REQ-ID].md
- 基本設計: docs/design/features/[REQ-ID]-logic.md
- API仕様: docs/api/openapi.yaml（該当エンドポイント: GET /api/xxx）
- 共通コンポーネント: docs/design/shared-components.md
<!-- 存在しないファイルは「未作成」と記載 -->

## インターフェース定義（ドメイン間連携がある場合）
- interfaces/ 参照: [ファイルパス]

## 備考・制約
<!-- 実装上の注意事項、仮採用した設計判断など -->
```

## 4. 下位層の起動と監視
- 依存関係のない下位層は**並列で起動**する
- 依存がある場合は先行タスクの**完了を確認してから**起動する
- 起動時に `logs/context/` のコンテキストファイルの内容を渡す
- [完了確認チェックリスト](completion-checklist.md) で各下位層の完了を判定する

### ブランチ同期ルール（並列実行時）

複数のPGタスクが並列で実行される場合、ブランチの分岐元を制御して import エラーを防止する：

- **独立タスク（相互に `depends_on` がない）:** 同じ main ブランチから各自 `feature/[REQ-ID]` を分岐してよい（並列実行可能）
- **依存タスク（`depends_on` で先行タスクが指定されている）:** 先行タスクが**mainにマージされた後**に `feature/[REQ-ID]` を分岐する。先行タスクのブランチから直接分岐しない（マージ前のコードに依存するとrebase地獄になる）
- **共通基盤タスク → 機能タスク:** 共通基盤タスクが全て main にマージされてから、機能タスクのブランチを作成する

```
main ─── 共通基盤マージ ─── AUTH マージ ──────────
              ↓                   ↓
         feature/共通基盤    feature/AUTH
                                  ↓ （AUTH マージ後に分岐）
                             feature/PROFILE
```
