# Policy: Review Standards（レビュー基準）

## 深刻度の定義

| 深刻度 | 説明 | 対応 |
|--------|------|------|
| CRITICAL | 本番障害・データ損失・セキュリティ脆弱性 | 必ず修正 |
| MAJOR | 機能不備・要件未充足・エッジケース未対応 | 必ず修正 |
| MINOR | コード品質・規約違反・可読性 | 信頼度80以上なら修正 |
| INFO | 改善提案・将来の懸念 | 記録のみ（修正不要） |

## 信頼度スコアの基準

| スコア | 説明 | 扱い |
|--------|------|------|
| 90-100 | コードで確認済みの確実な問題 | 修正依頼 |
| 80-89 | 高い確信があるが文脈依存の可能性 | 修正依頼 |
| 50-79 | 中程度の確信 | レポートに残すが修正依頼しない |
| 0-49 | 推測レベル（偽陽性の可能性が高い） | 報告しない |

**信頼度80以上の指摘のみ coder に修正依頼する。**

## finding_id（指摘追跡ID）

各指摘に一意のIDを付与し、イテレーション間で追跡する:

- 形式: `[CAT]-[FILE]-[LINE]-[SHORT_HASH]`
  - CAT: BUG / SEC / STD / PERF / DESIGN
  - FILE: ファイル名（拡張子除く）
  - LINE: 行番号
  - SHORT_HASH: 内容の4文字ハッシュ
- 例: `SEC-auth-42-a3f1`, `BUG-user-88-c7d2`
- 同一指摘が次のイテレーションでも残る場合、同じ finding_id を再利用する

## 指摘フォーマット

```markdown
#### [深刻度] 指摘タイトル（信頼度: XX/100）
- **finding_id:** [CAT]-[FILE]-[LINE]-[HASH]
- **ファイル:** path/to/file.ts:行番号
- **カテゴリ:** バグ / セキュリティ / 規約違反 / パフォーマンス / 設計
- **内容:** 問題の具体的な説明
- **修正案:** 推奨する修正方法
```

## REJECT条件（ブロッキング）

以下に該当する場合は CRITICAL/MAJOR として指摘:
- 新しい動作にテストがない
- バグ修正にリグレッションテストがない
- `any` 型の使用（TypeScript）
- フォールバック値の濫用（coding-quality.md 参照）
- What/How の説明コメント（Why のみ許可）
- 未使用コード・デッドコード
- 直接的なミューテーション（イミュータブルが期待される箇所）
- エラーの握りつぶし（空の catch）
- 未追跡の TODO コメント
- DRY 違反（3回以上の重複）
- リーキーアブストラクション
- リファクタリング後に残存する旧コード
- 関連フィールドのクロスバリデーション漏れ
