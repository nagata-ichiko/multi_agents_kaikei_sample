# Spec 執筆標準

全 Spec で統一するフォーマット・用語・ルールを定義する。
spec-all、spec-feature、draft-spec の Spec 執筆時に必ず参照すること。

## 1. 用語辞書

[Spec 統一基盤確認](./spec-unified-base.md) のチェックリスト1を参照。
glossary.md のテーブルには英語表記列を必須とする（i18n対応およびコード変数名との対応関係を明確にするため）。

## 2. Spec ファイルのセクション構成

全 Spec は以下のセクション順序を守る。該当しないセクションは省略可。

```markdown
# [REQ-ID] [機能名]

## 概要
1〜3行で機能の目的を説明する。

## 関連 Spec
- [REQ-XXX-NNN](./xxx.md): 関係の説明

## アクター
この機能を利用するユーザー種別。

## 機能要件

### [REQ-ID-NNN] [要件名]
**受入条件:**
- [ ] 具体的な条件（数値・境界値を明記）

## 画面遷移図
（UI を持つ機能のみ。screen-transition-diagram.md ルールに従う）

## ビジネスロジック
（複雑なロジックがある場合。Mermaid 図で視覚化）

## API エンドポイント
| メソッド | パス | 説明 | コード根拠 |
|----------|------|------|-----------|

## データモデル
関連するテーブルとカラムの説明。

## 非機能要件
パフォーマンス、セキュリティ等。

## コード根拠
| 項目 | ファイル:行 |
|------|-----------|
```

## 3. 機能要件の記述ガイダンス

以下の3観点は受入条件の品質に直結する。曖昧な記述は下流（詳細設計・実装）でのバグの原因になるため、具体値を必ず明記すること。

### バリデーションルール
- 受入条件に具体的なパターンで記述する（文字数、形式、許容値）
- 「適切な」「正しい形式」等の曖昧な記述は禁止。数値・正規表現・許容値リストで示す
- 例:
  - ○ `- [ ] メールアドレスは RFC 5322 準拠。最大254文字`
  - ○ `- [ ] パスワードは8〜72文字。英大文字・英小文字・数字を各1文字以上含む`
  - × `- [ ] 適切なバリデーションがある`

### エラーケース
- 主要な異常系を受入条件に含める。正常系だけの受入条件は不完全とみなす
- 例:
  - ○ `- [ ] メールアドレスが既に登録済みの場合、「このメールアドレスは既に使用されています」と表示する`
  - ○ `- [ ] 同時に同じリソースを更新した場合、後からの更新は「他のユーザーが更新しました」エラーを返す`
  - × エラーケースの受入条件がゼロ

### 権限・アクセス制御
- ロール別の振る舞いを明記する。アクターが複数ある場合は必須
- 例:
  - ○ `- [ ] 管理者は全ユーザーの注文を閲覧できる。一般ユーザーは自分の注文のみ閲覧できる`
  - ○ `- [ ] 下書き状態の記事は作成者と管理者のみ閲覧できる`
  - × ロールが複数あるのに権限の違いが未記載

## 4. 記述ルール

### 文体
- 「〜する」「〜できる」の体言止め・動詞終止形で統一
- 「〜と思われる」「一般的に〜」「おそらく〜」は禁止

### コード根拠
- 全ての機能記述にコード根拠を付与する。根拠がない記述は書かない
- **パス形式:** コードリポのルートからの相対パス（例: `src/auth/signup.ts:42`）
  - 行番号で示す: `src/auth/signup.ts:42` （基本形）
  - 行範囲で示す: `src/auth/signup.ts:42-58`
  - メソッド名で示す: `src/auth/signup.ts:signup()` （行番号が頻繁に変わる場合のみ）
- **記載方式:** 以下のいずれかを使い分ける
  - **インライン方式**: 各記述の直後に `（src/auth/signup.ts:42）` のように付記する。個別の要件・ルールの根拠を明示する場合に使う
  - **表方式**: Specの末尾「コード根拠」セクションに一覧表でまとめる。ファイル一覧としての俯瞰性が必要な場合に使う
  - 1つのSpec内で両方を併用してよい（インラインで個別根拠を示し、末尾表で全体をまとめる）

### 受入条件
- チェックリスト形式（`- [ ]`）で統一
- 数値・境界値を明記する（「適切な」「十分な」は禁止）
- 例: `- [ ] ファイルサイズ上限は 10MB` ○ / `- [ ] 適切なサイズ制限がある` ×

### REQ-ID
- 形式: `REQ-[機能カテゴリ]-[3桁連番]`
- 機能カテゴリは英大文字（例: AUTH, PROJECT, CONTRACT, POINT）
- 一度採番したら変更しない

### 相互参照
- 他の Spec を参照する場合は `[REQ-XXX-NNN](./xxx.md)` の相対リンクを使う
- 共通ロジックは各機能の `docs/design/features/*-logic.md` に記載し、機能横断の共通ロジックは該当する横断設計書に集約する

### GitHub Issue 参照
- Issue 番号は必ずクリック可能なリンクにする。`#NNN` のプレーンテキストは禁止
- 形式: `[#NNN](https://github.com/{org}/{repo}/issues/NNN)`
- リポジトリは `git remote get-url origin` から判定する
- `!!! warning` の要確認ボックスのタイトル・本文、変更履歴、本文中いずれも同じ形式で統一する
- 例:
  - ○ `!!! warning "要確認: [#158](https://github.com/org/repo/issues/158)"`
  - ○ `初版作成（[#41](https://github.com/org/repo/issues/41) 部署管理の要件定義）`
  - × `!!! warning "要確認: #158"`
  - × `初版作成（#41 部署管理の要件定義）`
