# Policy: Loop Protocol（レビュー→修正ループ監視プロトコル）

## 概要
review→fix ループの品質と効率を管理する。無限ループ・堂々巡りを防止し、必要に応じてスーパーバイザーに判断を委任する。

## イテレーション管理

### カウンター
- 最大3イテレーション（ハードリミット）
- reviewer は各レビューに `## レビュー結果（イテレーション N/3）` ヘッダーを付与する
- coder は修正時に `## 修正対応（イテレーション N/3）` ヘッダーを付与する

### finding_id による追跡
各指摘に finding_id を付与し、イテレーション間で追跡する（形式は review-standards.md 参照）。

イテレーション2以降、reviewer は前回の finding_id リストを参照し各指摘を分類する:
- **resolved**: 前回指摘が修正済み
- **persists**: 前回と同じ finding_id が未修正のまま残存
- **new**: 今回新たに発見された指摘
- **regressed**: 前回 resolved だったが再発した指摘

## 収束/発散の判定

| シグナル | 判定 | アクション |
|---------|------|----------|
| finding数が減少、persists=0 | 健全（収束中） | 通常続行 |
| finding数が減少、persists≧1 | 概ね健全 | persists の指摘に代替アプローチを提案 |
| 同一 finding_id が2回以上 persists | 停滞 | イテレーション2: reviewer が代替アプローチ提案。イテレーション3: スーパーバイザー判定 |
| 新規 finding が前回より増加 | 発散（スコープクリープまたはリグレッション） | 即座にスーパーバイザー判定 |
| regressed finding あり | リグレッション | CRITICAL扱いで即修正依頼 |

## スーパーバイザー介入（イテレーション3到達時）

イテレーション3で未解決の指摘がある場合、reviewer はスーパーバイザーとして以下を判定する:

### 判定基準

| 状態 | 判定 | アクション |
|------|------|----------|
| persists が MINOR のみ、かつ1-2件 | **例外付き承認** | 残存指摘を「既知の制限事項」としてレポートに記録し、承認する |
| persists に CRITICAL/MAJOR あり | **中止** | 根本原因分析を実施し、ユーザーにエスカレーション |
| 発散している（新規指摘が増加中） | **中止** | 設計の見直しが必要。ユーザーにエスカレーション |

### スーパーバイザー判定レポート

以下のフォーマットで記録する:

```markdown
## スーパーバイザー判定

### イテレーション履歴
| イテレーション | finding数 | resolved | persists | new | 傾向 |
|--------------|----------|----------|----------|-----|------|
| 1 | X | - | - | X | 初回 |
| 2 | Y | A | B | C | 収束/停滞/発散 |
| 3 | Z | D | E | F | 収束/停滞/発散 |

### 残存指摘の分析
| finding_id | 深刻度 | 概要 | persists回数 | 根本原因 |

### 判定
- **例外付き承認** / **中止**

### 理由
[判定の具体的根拠]
```

## coder への修正指示フォーマット

イテレーション2以降、reviewer が coder に修正を依頼する際は以下を含める:

```markdown
## 修正依頼（イテレーション N/3）

### 前回からの変化
- resolved: X件（finding_id一覧）
- persists: Y件（finding_id一覧 + 今回の代替アプローチ提案）
- new: Z件

### 修正必須（persists + new の CRITICAL/MAJOR）
[指摘詳細]

### 注意
- persists の指摘は、前回と同じアプローチではなく代替案を試すこと
- 前回 resolved を壊さないよう注意（リグレッション防止）
```
