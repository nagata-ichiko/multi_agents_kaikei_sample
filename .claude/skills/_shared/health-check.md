# プロジェクトヘルスチェック

プロジェクトの健全性を定量的に把握するための手順。
セッション開始時や実装前の状況確認に使用する。

## チェック項目と算出方法

### 1. Spec充足率
- `docs/requirements/features/` 内のSpecファイル数をカウント
- spec-map.yml のREQ-ID数をカウント
- **充足率** = spec-map.yml に登録済みのREQ-ID数 / Specファイル数 × 100%
- 100%未満 → 未実装のSpecがある

### 2. Spec-Code一致率
- spec-map.yml の全エントリについて `confirmed` と `spec_version` を比較
- **一致率** = confirmed == spec_version のエントリ数 / 全エントリ数 × 100%
- 100%未満 → Spec変更後にコードが追従していない箇所がある

### 3. テストカバレッジ
- spec-map.yml で `tests:` が空のREQ-IDをカウント
- **テスト未カバー率** = tests未登録のREQ-ID数 / 全REQ-ID数 × 100%
- 0%が理想

### 4. 依存グラフ健全性
- `docs/design/dependency-graph.md` の循環依存セクションを確認
- 循環依存がゼロなら健全

### 5. 設計書鮮度
- `docs/design/` 配下のファイルの最終更新日を確認
- 最新コミット日と30日以上離れているファイルがあれば警告

## ヘルスレポートのフォーマット

```markdown
## プロジェクトヘルス（YYYY-MM-DD）

| 指標 | 値 | 状態 |
|------|-----|------|
| Spec充足率 | XX% | OK / 要対応 |
| Spec-Code一致率 | XX% | OK / 要対応 |
| テストカバー率 | XX% | OK / 要対応 |
| 循環依存 | 0件 | OK / 要対応 |
| 設計書鮮度 | 全て30日以内 | OK / 要対応 |

### 要対応項目
- [ ] REQ-XXX-NNN: confirmed(1) < spec_version(2) → コード追従が必要
- [ ] REQ-YYY-NNN: テスト未登録 → gen-tests で生成
```

## 判定基準

| 指標 | OK | 要対応 |
|------|-----|--------|
| Spec充足率 | 100% | 100%未満 |
| Spec-Code一致率 | 100% | 100%未満 |
| テストカバー率 | 80%以上 | 80%未満 |
| 循環依存 | 0件 | 1件以上 |
| 設計書鮮度 | 全て30日以内 | 30日超のファイルあり |
