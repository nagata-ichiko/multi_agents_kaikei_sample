# 共通仕上げ手順

コードを変更するスキル（implement-spec, revise-spec, apply-design）はこの手順で仕上げる。

## 0. ヘルスチェック（実装前）
実装開始前に [ヘルスチェック](health-check.md) の「Spec-Code一致率」を確認する。
対象REQ-IDの `confirmed < spec_version` がある場合は、先にコード追従を行うか判断する。

## 1. テスト実行
- ユニットテストを実行して全てパスすることを確認
- UIを含む変更の場合、E2Eテスト（CLAUDE.mdの`test_e2e`参照）も実行

## 2. 人間確認用テストケースExcel追記
該当する項目がある場合のみ、コードリポの `manual-test-cases.xlsx` に追記する（なければ新規作成）。
Excelライブラリ（Python=openpyxl, Node.js=exceljs, Ruby=caxlsx 等。tech-stack-guide.md の「帳票/ファイル生成」参照）を使用。既存シートの末尾に行を追加する。該当項目がなければスキップ。

**対象となる分類：**
- OAuth確認：外部OAuth（X, Google等）の実ログインが関わる場合
- 決済確認：決済処理が関わる場合
- メール/SMS確認：メール・SMS送信が関わる場合
- デザイン確認：UIの見た目が変わる場合
- レスポンシブ確認：UIが変わる場合（実機での操作感）
- 文言確認：ユーザー向けテキストが変わる場合
- 外部連携確認：外部API（X API, Stripe等）の実連携が関わる場合

**Excelの列：**
| Spec ID | 分類 | テストケース | 手順 | 期待結果 | 確認者 | 結果 | 備考 |

「確認者」「結果」「備考」は空欄（人間が記入する）。

## 3. 設計書⇔コード一致性検証
実装完了後、以下のチェックリストで設計書とコードの一致を検証する。該当するファイルが存在する項目のみ実施する。

### 3a. API一致性（openapi.yaml が存在する場合）
- [ ] 実装したAPIエンドポイントのパス・メソッドが openapi.yaml の定義と一致する
- [ ] リクエストボディのフィールド名・型が openapi.yaml の requestBody スキーマと一致する
- [ ] レスポンスのフィールド名・型・ステータスコードが openapi.yaml の responses 定義と一致する
- [ ] エラーレスポンスの形式が openapi.yaml の error スキーマと一致する

### 3b. DB一致性（db-schema.md が存在する場合）
- [ ] 作成・変更したテーブル名・カラム名が db-schema.md の定義と一致する
- [ ] カラムの型・制約（NOT NULL, UNIQUE, DEFAULT等）がマイグレーションコードと一致する
- [ ] インデックス定義が db-schema.md の記載と一致する

### 3c. エラーコード一致性（error-codes.md が存在する場合）
- [ ] 実装で使用しているエラーコード・メッセージが error-codes.md に定義されている
- [ ] 新規追加したエラーコードが error-codes.md に追記されている

### 3d. 要件一致性（常に実施）
- [ ] 要件定義書の受入条件が全てテストケースでカバーされている
- [ ] feature-logic.md のバリデーションルールが実装に反映されている

### 3e. 残存参照チェック（常に実施）
- [ ] 変更・削除したファイルパスやUI要素名で設計書をgrepし、古い参照が残っていないか確認する
- [ ] 変更前の旧名称でコードベース全体をgrepし、旧記述が残っていないか確認する

### 3f. spec-map.yml ファイルパス実在性検証（常に実施）
- [ ] spec-map.yml の対象REQ-IDの implementation/tests 全パスについて、ファイルが実際に存在するか確認する
- [ ] 削除したファイルのエントリが spec-map.yml に残っていれば削除する
- [ ] 新規作成したファイルが spec-map.yml に未登録であれば追加する

画面遷移図の更新が必要な場合は [画面遷移図ルール](screen-transition-diagram.md) に従う。

**不一致を発見した場合:** 設計書とコードのどちらが正しいか判断し、同じコミットで両方を修正する。判断に迷う場合はユーザーに確認する。

## 4. 依存グラフの更新
`docs/design/dependency-graph.md` が存在し、かつ今回の変更でSpec間の依存関係に影響がある場合（新しいSpecへの依存追加、既存依存の削除等）:
- spec-map.yml の `depends_on` フィールドを更新する
- dependency-graph.md のMermaid図とマトリクスを更新する

## 5. セッション状態の保存
`tasks/session-state.md` の「完了済みステップ」を更新し、コミット前の状態を記録する。
これにより、セッションが途中で切れても前回の作業状態を把握できる。

## 5b. shared-components.md 更新（並列実行時の排他制御）

新たに共通コンポーネントを追加・変更した場合、`docs/design/shared-components.md` を更新する。

**並列実行時のルール（orchestrate 経由で複数PGが同時稼働している場合）:**
- 更新前に `shared-components.md` の**最新版を再読**する（他のPGが先に更新している可能性がある）
- 既存エントリを上書きせず、**末尾に追記**する
- 他のPGが追加したエントリを削除・変更しない
- コンフリクトが発生した場合は、両方のエントリを残してコミットし、SuperPMの統合確認フェーズで整理する

## 5c. spec-map.yml 更新（並列実行時の排他制御）

**並列実行時のルール（orchestrate 経由で複数PGが同時稼働している場合）:**
- 更新前に `spec-map.yml` の**最新版を再読**する（他のPGが先に更新している可能性がある）
- 自タスクの REQ-ID のエントリのみ追加・変更する。**他の REQ-ID のエントリは変更しない**
- 異なる REQ-ID のエントリは YAML のトップレベルキーが異なるためコンフリクトしにくいが、コンフリクトが発生した場合は両方のエントリを残してコミットする
- SuperPMの統合確認フェーズで最終的な整合性を検証する

## 6. コミット
変更をコミットする。コミットメッセージはCLAUDE.mdのブランチ戦略に従う。

## 7. テンプレートフィードバックチェック
実装中に以下に該当する修正をしたか振り返る：
- 非推奨APIを踏んで修正した（例：Next.jsのparams Promise化）
- CI設定の不備を修正した
- 環境依存の問題を回避した（例：Supabase CLIの出力フォーマット変更）
- スキルの手順不足で想定外の動きをした

該当するものがあれば：
1. 「この問題は他のプロジェクトでも起きる可能性があります。テンプレートリポにフィードバックしますか？」と提案する
2. .claude/CLAUDE.mdの「テンプレートリポ」にURLが未設定の場合は、先にURL設定を促す
3. ユーザーが同意したら feedback-template スキルを実行する

## 8. エラー発生時の参照
実装中に問題が発生した場合は [エラーリカバリーパターン集](error-recovery.md) を参照する。
特にテスト失敗（パターン1）、CI失敗（パターン2）、Spec⇔コード不整合（パターン3）は頻出。
