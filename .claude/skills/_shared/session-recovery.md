# セッション復帰プロトコル

セッション断絶後に前回の作業を安全に再開するための手順。

## セッション状態の保存

スキル実行中、以下のタイミングで `tasks/session-state.md` を更新する:
- スキルのフェーズが1つ完了するたび
- サブエージェントの起動/完了時
- コミット完了時

### session-state.md のフォーマット

```markdown
# セッション状態

## 最終更新
- 日時: YYYY-MM-DD HH:MM
- 実行中スキル: [スキル名]
- 現在のフェーズ: [フェーズ番号/名前]

## 完了済みステップ
- [x] ステップ1の内容
- [x] ステップ2の内容

## 未完了ステップ
- [ ] ステップ3の内容
- [ ] ステップ4の内容

## コンテキスト
- 対象Spec: [REQ-ID]
- 作業ブランチ: [ブランチ名]
- 関連ログ: [logs/context/ファイル名]

## 中断時の注意事項
<!-- 再開時に知っておくべき情報（途中の判断、未解決の問題等） -->
```

## セッション復帰手順

新セッション開始時に以下を確認する:

1. `tasks/session-state.md` が存在するか確認
2. 存在する場合:
   - 内容を読み、前回の作業状態を把握する
   - 「前回の続きから再開しますか？」とユーザーに提案する
   - ユーザーが同意 → 未完了ステップから再開
   - ユーザーが拒否 → session-state.md をクリアして新規作業開始
3. 存在しない場合: 通常通り新規作業を開始

## 復帰時の検証

再開前に以下を確認する:
- 作業ブランチに未コミットの変更がないか（`git status`）
- spec-map.yml に中途半端な状態がないか
- `logs/context/` に前回配布したコンテキストファイルがあるか
- `tasks/todo.md` の進捗が session-state.md と一致しているか

## 長時間タスクでの保存ポイント

### spec-all
- フェーズ0完了時（glossary作成後）
- 各ドメインのSpec生成完了時
- 統合レビュー完了時

### implement-spec / revise-spec
- 設計書更新完了時
- コード実装完了時（テスト前）
- テスト完了時（コミット前）

### orchestrate
- タスク分解完了時
- 各ドメインPM完了時
- 全体統合完了時
