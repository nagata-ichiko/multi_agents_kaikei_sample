# Spec 統一性レビュー

spec-all および spec-feature の完了時に実行する横断チェック。
レビューエージェント（サブエージェント）が全既存 Spec を読み、統一性を検証する。

## 起動タイミング

| スキル | タイミング |
|--------|-----------|
| spec-all | 各バッチ完了ごと + 全体完了時 |
| spec-feature | 個別 Spec 書き出し後 |

## レビューエージェントへの指示テンプレート

```
あなたは Spec 統一性レビュアーです。
以下の Spec ファイル群を読み、統一性の問題を報告してください。

## 対象
- docs/requirements/overview.md
- docs/requirements/features/ 配下の全 .md ファイル
- docs/design/architecture.md
- docs/design/db-design.md
- docs/design/api-spec.md（外部連携仕様書。存在する場合のみ）
- docs/design/screen-flow.md
- spec-map.yml（REQ-ID ごとに実装ファイル・テストファイルのパスと spec_version/confirmed を管理するファイル。init-spec で空テンプレートが作成され、spec-feature/spec-all でエントリが追加される）

## 今回の新規/更新 Spec
[ここに今回書いた Spec のファイルパスを列挙]

## チェック項目

### 1. 用語統一
- 同じ概念に異なる名称を使っていないか
  例: 「案件」vs「プロジェクト」vs「求人」、「ユーザー」vs「エンジニア」vs「求職者」
- モデル名・テーブル名との対応が一貫しているか
- 用語のブレを見つけたら、全Specでどの表記が多いかを集計し、統一案を提示する

### 2. REQ-ID 整合性
- REQ-ID に重複がないか
- overview.md の機能一覧と features/ 配下のファイルが一致しているか
- REQ-ID の命名規則（REQ-[機能]-[連番]）に従っているか

### 3. 相互参照の整合性
- Spec 間で参照している機能が実在するか（存在しない REQ-ID を参照していないか）
- 「○○機能で定義」と書かれた先に実際にその定義があるか

### 4. 設計書との整合性
- Spec に記載された DB テーブルが db-design.md に存在するか
- Spec に記載された API エンドポイントが openapi.yaml（APIの Single Source of Truth）に存在するか
- Spec に記載された画面が screen-flow.md に存在するか

### 5. フォーマット統一
- 全 Spec のセクション構成が統一されているか（見出しレベル、セクション順序）
- 受入条件がチェックリスト形式で統一されているか
- コード根拠（ファイル名:行番号）の記載形式が統一されているか

### 6. 境界の明確さ
- 複数の Spec にまたがる機能が、どちらの Spec で定義されるか明確か
- 重複した記述がないか（同じビジネスルールが複数 Spec に書かれていないか）

### 7. 機能横断の画面遷移
- 異なるSpec間をまたぐ画面遷移（例: ユーザー管理 → ポイント管理への跨ぎ遷移）が screen-flow.md に反映されているか
- 個別Specの画面遷移図に記載された遷移先が、遷移先Specの画面として実在するか
- screen-flow.md の統合画面遷移図が、全個別Specの画面遷移を網羅しているか

### 8. OVERVIEW との非機能要件同期
- 個別Specに記載された非機能要件がOVERVIEWと矛盾していないか（例: 個別Specで「SLA 99.9%」、OVERVIEWで「99.5%」）
- 個別Specで新しい外部サービス依存・技術スタック・認証方式が追加された場合、OVERVIEWにも反映されているか

## 出力形式

### 統一性レポート

#### 問題なし
- [チェック項目]: OK

#### 要修正（自動修正可能）
- [ファイル]: [問題内容] → [修正案]

#### 要確認（人間の判断が必要）
- [ファイル]: [問題内容] → [選択肢A] or [選択肢B]

#### 用語統一テーブル（ブレが見つかった場合）
| 概念 | 現状の表記ゆれ | 推奨表記 | 理由 |
|------|--------------|---------|------|

## 厳守事項
- 全ファイルを実際に読んでチェックする（ファイル名だけで判断しない）
- 問題がなければ「OK」と報告する（無理に問題を見つけようとしない）
- 修正案は具体的に（どのファイルの何行目をどう変えるか）
```

## メインエージェントの対応

レビューエージェントの報告を受けて：
1. **自動修正可能** → メインが即座に修正
2. **要確認** → ユーザーに選択肢を提示して判断を仰ぐ
3. 修正後、用語統一テーブルがあれば `docs/requirements/glossary.md` に反映する
