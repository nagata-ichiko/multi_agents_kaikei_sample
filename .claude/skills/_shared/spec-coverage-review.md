# 仕様カバレッジレビュー

コードから設計書を生成した後、**生成エージェントとは別のレビューエージェント**がコードと設計書を突合し、仕様の書き漏らしを検出する。

## 適用タイミング

| スキル | タイミング |
|--------|-----------|
| spec-all | 全バッチ完了後（フェーズ6の一部として） |
| detail-design | 全設計書生成後（フェーズ3の整合性チェック後） |

## レビューエージェントへの指示テンプレート

```
あなたは仕様カバレッジレビュアーです。
コードに存在する仕様が設計書に漏れなく記載されているかを検証してください。

## 対象
- コードリポ: [CLAUDE.mdのプロジェクト構成から取得]
- 設計書: docs/requirements/features/*.md, docs/design/features/*.md, docs/api/openapi.yaml, docs/design/db-schema.md

## チェック観点

### 1. バリデーション漏れ
コード内のバリデーション（CHECK制約、フォームバリデーション、APIパラメータ検証）が
設計書に記載されているか。

チェック方法:
- DBマイグレーションのCHECK制約 → db-schema.mdに記載があるか
- APIルートのパラメータ検証（必須チェック、型チェック、範囲チェック）→ openapi.yamlに記載があるか
- フロントエンドのフォームバリデーション → 機能別設計書の画面構成に記載があるか

### 2. エラーハンドリング漏れ
コード内のエラーレスポンス（try-catch、HTTPステータスコード返却）が
設計書に記載されているか。

チェック方法:
- 各route.tsのNextResponse.json({error:...}, {status:...})
  → openapi.yamlの該当エンドポイントのresponsesに記載があるか

### 3. 条件分岐漏れ
コード内のif/switch等の分岐ロジックが設計書に記載されているか。

チェック方法:
- 権限チェック（staff検証、ロール検証）→ 機能別設計書のビジネスルールに記載があるか
- 状態による分岐（status=published のみ許可等）→ 機能別設計書に記載があるか

### 4. DBトリガー・関数漏れ
DBのトリガー・関数がdb-schema.mdに記載されているか。

チェック方法:
- マイグレーション内のCREATE FUNCTION / CREATE TRIGGER → db-schema.mdに記載があるか

### 5. 外部API呼び出し漏れ
コード内の外部API呼び出しがexternal-integration.mdに記載されているか。

## 出力形式

### カバレッジサマリー

| チェック観点 | コード件数 | 設計書記載数 | カバレッジ | 漏れ件数 |
|------------|----------|-----------|----------|---------|

### 漏れ一覧

| # | 観点 | コード箇所 | 内容 | 記載すべき設計書 |
|---|------|----------|------|---------------|

## 厳守事項
- コードを実際に読んで検証する（設計書だけ見て判断しない）
- 漏れがない場合は「カバレッジ100%」と報告する（無理に問題を見つけない）
- 各漏れに対して、どの設計書のどのセクションに追記すべきかを具体的に提案する
```

## メインエージェントの対応

レビューエージェントの報告を受けて：
1. カバレッジ100% → そのまま完了
2. 漏れあり → 該当する設計書に追記してコミット
3. 追記後、再度レビューは不要（1回で十分。過剰品質を避ける）
