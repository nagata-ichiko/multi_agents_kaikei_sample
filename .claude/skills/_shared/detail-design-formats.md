# 詳細設計フォーマット定義

フェーズ2で生成する各設計書のフォーマット、ルール、コミットメッセージを定義する。

## フェーズ 2: 設計書執筆（メインが一元管理）

フェーズ1の情報収集結果をもとに、**メインエージェントが設計書を執筆する**。
PM オーケストレーション共通パターン（`.claude/rules/pm-orchestration.md` で自動ロード） の「1ファイルずつコミットパターン」「コンテキスト圧縮対策」に従う。

**フォーマットは両モード共通。** 以下のルールのみモードで異なる：

| ルール | コード分析モード | 設計書ファーストモード |
|--------|----------------|-------------------|
| コード根拠 | 必須（ファイル名:行番号） | 不要（代わりに関連Specの REQ-ID を記載） |
| 値の出典 | コードから正確に抽出 | 要件定義書 + ユーザー回答。未確定値には「※ 未確定」と注記 |
| リクエスト/レスポンス例 | 実際のコードから生成 | 要件定義書のデータモデルから設計（ベストプラクティスに基づく） |
| バリデーション | モデルの validates 定義を転記 | 要件定義書の受入条件から導出 |

### 2.1 DB詳細スキーマ (`docs/design/db-schema.md`)

**フォーマット:**

```markdown
# DB詳細スキーマ

## テーブル一覧

| # | テーブル名 | 説明 | レコード規模 |
|---|-----------|------|-------------|

## 詳細定義

### [テーブル名]

**概要:** [1行説明]
**関連Spec:** [REQ-XXX](../requirements/features/xxx.md)

| カラム | 型 | NULL | デフォルト | 説明 |
|--------|-----|------|-----------|------|
| id | bigint | NO | auto_increment | 主キー |
| name | varchar(255) | NO | - | 名前 |

**インデックス:**
| 名前 | カラム | ユニーク | 条件 |
|------|--------|---------|------|

**外部キー:**
| カラム | 参照先 | ON DELETE |
|--------|--------|----------|
```

**ルール:**
- db/schema.rb（またはマイグレーション）から**全カラムを漏れなく**記載する
- 型は DB ネイティブ型で記載（`string` ではなく `varchar(255)`、`integer` ではなく `int(11)` 等）
- レコード規模は推定で「少量（〜100）」「中量（〜10,000）」「大量（10,000〜）」のいずれか
- 既存の db-design.md との整合性を確認し、矛盾があれば db-design.md も修正する

→ コミット: `docs: DB詳細スキーマを追加 (db-schema.md)`

### 2.2 OpenAPI 完全版 (`docs/api/openapi.yaml`)

既存の openapi.yaml を精緻化する。

**追加する内容:**
- `components/schemas/` に全リクエスト/レスポンスのスキーマ定義
- 各パスに `requestBody` と `responses` の詳細定義
- `components/parameters/` に共通パラメータ（ページネーション等）

**ルール:**
- **APIの正（Single Source of Truth）。** API仕様はここにのみ定義する
- OpenAPI 3.1 準拠
- `$ref` を活用してスキーマの重複を避ける
- 既存のパス定義は上書きではなく拡充する

→ コミット: `docs: OpenAPI仕様を完全版に拡充 (openapi.yaml)`

- **認証:** [API Key / OAuth2 / Basic Auth 等]
- **コード根拠:** [ファイル名:行番号]

### エンドポイント
| 用途 | メソッド | URL | 認証 |
|------|---------|-----|------|

### リクエスト/レスポンス

#### [API名]
**リクエスト:**
\```json
{ ... }
\```

**レスポンス:**
\```json
{ ... }
\```

### エラーハンドリング
| エラーコード | 原因 | 対処 | コード根拠 |

- 実際のコードで呼び出している外部APIのみ記載する

### 2.15 機能別設計書 (`docs/design/features/*.md` + `*-logic.md`)

**各機能を画面・UI設計とロジック設計の2ファイルに分割して記載する。**
実装担当者はこの2ファイル + 横断設計書で1機能を実装できる。

- **画面・UI設計:** テンプレート（`docs/templates/phase2/feature-design.md`）に従う
- **ロジック設計:** テンプレート（`docs/templates/phase2/feature-logic.md`）に従う

**生成単位:** `docs/requirements/features/` 内の要件定義書1つにつき2ファイル

**生成先:**
- `docs/design/features/[feature-name].md` — 画面・UI設計
- `docs/design/features/[feature-name]-logic.md` — ロジック設計
- ファイル名は要件定義書のファイル名に合わせる（例: `auth.md` → `auth.md` + `auth-logic.md`）

**画面・UI設計書（feature-design.md）に含めるセクション:**
- **概要:** 機能の目的（1-2行）、デザインカンプ
- **画面設計:** レイアウト、入力要素、操作ボタン、データバインディング、画面状態、レスポンシブ
- **画面遷移:** 機能内の遷移図・パラメータ受け渡し
- **共通コンポーネント:** 利用する既存コンポーネント・新規提案
- **ページ固有コンポーネント:** コンポーネント構成図・一覧
- **コード根拠:** UI関連ファイルのみ

**ロジック設計書（feature-logic.md）に含めるセクション:**
- **使用APIエンドポイント:** 一覧（詳細はopenapi.yaml参照）、呼び出し順序、権限チェック
- **使用データモデル:** テーブル一覧、削除時の関連データ
- **ビジネスロジック:** 処理フロー（sequenceDiagram）、状態遷移、バリデーション、一覧仕様、代替/例外フロー

**1情報=1箇所の原則に基づく横断設計書との使い分け:**
- APIリクエスト/レスポンス/エラー → **openapi.yaml が正。** features/*.md には使用エンドポイント一覧（メソッド+パス+概要）のみ記載
- DBカラム定義 → **db-schema.md が正。** features/*.md には使用テーブル名のみ記載
- Enum/定数の許容値 → **master-data.md が正。** features/*.md にはコピーしない
- 外部API仕様 → **external-integration.md が正。** features/*.md では「外部連携あり」とリンクのみ
- **処理フロー・画面構成・ビジネスルールは features/*.md が正**（横断設計書には書かない）

**生成ルール:**
- 横断設計書の生成が**全て完了した後**に生成する
- コード分析モード: コード根拠（ファイル名:行番号）を必ず付与する
- 設計書ファーストモード: 関連Spec（REQ-ID）を記載し、未確定値には「※ 未確定」と注記する
- 状態遷移がある機能: 遷移×権限マトリクス・副作用テーブル・同時操作方針の3点セットを含める

**サブエージェントによる並列生成:**
- 機能数が多い場合（5機能以上）、サブエージェントで並列生成する
- 1サブエージェント = 最大3機能を担当（画面・UI + ロジックの2ファイルをセットで生成）
- 各サブエージェントには横断設計書の参照パスと、担当する要件定義書のパスを渡す

→ コミット: `docs: 機能別設計書を追加 (features/)`
