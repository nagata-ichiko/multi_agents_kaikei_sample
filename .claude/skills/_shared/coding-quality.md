# Policy: Coding Quality（コーディング品質ルール）

## 基本原則

- **Simple > Easy**: 理解しやすさ > 書きやすさ。暗黙の挙動より明示的な制御
- **DRY**: 同一ロジックは1箇所に。3回以上の重複は共通化する
- **Fail Fast**: エラーは早期に検出し、曖昧なフォールバックを作らない
- **Why コメントのみ**: What/How のコメントは不要。「なぜこの判断か」だけ書く
- **関数は〜30行**: 1関数1責務。超過したら分割を検討
- **ファイルは〜300行**: 超過したら責務分割を検討
- **Boy Scout Rule**: 触ったファイルは少し良くして返す

## 禁止パターン（フォールバック濫用）

以下のパターンは**明示的な理由なしに使用禁止**:

| 禁止 | 正解 |
|------|------|
| `user?.id ?? 'unknown'` | 呼び出し元で user の存在を保証する |
| 全呼び出し元が省略するデフォルト引数 | 必須パラメータにする |
| null合体で到達不能なデフォルト値 | パス検証で早期エラー |
| 空の catch ブロック | エラーを上位に伝播、または明示的にログ |
| 条件に該当しない場合の黙殺（silent skip） | 明示的にログまたはエラー |

**例外**: 外部入力（ユーザー入力、外部API）のバリデーションでのフォールバックは許可。その場合はコメントで理由を記載。

## 判断ヒューリスティクス（設計情報不足時）

| 不足情報 | デフォルト判断 |
|---------|--------------|
| バリデーション | 受入条件から導出。なければ一般的制約を仮採用し `// TODO: 仮採用` と注記 |
| エラーハンドリング | 既存コードのパターンに合わせる。なければ標準HTTPステータス |
| 権限制御 | アクターが1種類のみなら認証チェックのみ |
| 状態遷移 | 明記されていない遷移は**禁止**として実装（許可リスト方式） |
| 削除方式 | db-schema.md 参照。なければ論理削除（`deleted_at`） |
| 同時操作 | logic.md 参照。なければ楽観的ロック（`updated_at` 比較） |
| リトライ方針 | logic.md 参照。なければ最大3回・指数バックオフ（2^n × 100ms） |
| ログ出力 | 既存パターンに合わせる。なければエラー時のみ `error` レベル |
| キャッシュ | 明示的指示がなければキャッシュしない |

**仮採用した判断は全て影響範囲レポートの「仮採用した設計判断」セクションに記録すること。**

## 抽象化ルール

- 条件分岐が3つ以上 → パターンマップへの置換を検討
- オーケストレーション関数は同一抽象レベルの呼び出しで構成する（低レベル操作を混ぜない）
- インターフェースは使用者視点で設計する（設定と実行を分離）
- 汎用レイヤーに特定実装が混入している場合 → リーキーアブストラクション（修正対象）

## 状態管理
- 状態は使用箇所の最も近い場所に閉じ込める
- 子→親への通知はイベント経由（直接的な親の状態変更は禁止）
- エラーハンドリングは上位レイヤー（Controller/Handler）に集約する
