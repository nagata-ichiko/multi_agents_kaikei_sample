# フェーズ 1: 情報収集

モードによってフェーズ1の動作が異なる。

## コード分析モード: フェーズ 1A — 並列コード分析

### 1A.1 サブエージェントの起動

成果物ごとに特化したサブエージェントを**並列起動**する。
PM オーケストレーション共通パターン（`.claude/rules/pm-orchestration.md` で自動ロード） の「大規模プロジェクトの分割基準」に従い、閾値を超える場合は分割する。

以下の3つのサブエージェントを起動する。各サブエージェントには PM オーケストレーション共通パターン（自動ロード済み） の「サブエージェント運用ルール > 起動時の共通指示」を必ず含める。

**サブエージェント一覧:**

| # | 名称 | 対象ファイル | 抽出内容 | 対応する成果物 |
|---|------|------------|---------|--------------|
| 1 | DB スキーマ分析 | スキーマ定義（例: Rails=`db/schema.rb`, Prisma=`schema.prisma`）、モデル層（tech-stack-guide.md参照） | 全テーブル×全カラム（名前・型・デフォルト・NULL・コメント）、インデックス、外部キー、Enum定義、バリデーション制約値 | db-schema.md, master-data.md |
| 2 | API・ロジック・画面分析 | コントローラ層 + シリアライザ + モデル層 + サービス層 + ページ + コンポーネント（tech-stack-guide.md参照） | リクエスト/レスポンス構造、業務フロー、状態遷移、フォーム要素、画面構成 | features/*.md, openapi.yaml |
| 3 | 外部連携・メール・環境分析 | パッケージ定義（tech-stack-guide.md参照）、設定ファイル、メーラー、docker-compose.yml、.env.example、README.md | 外部API連携詳細、メーラー（トリガー・宛先・件名・本文変数）、全ENV参照箇所、依存サービス接続設定、セットアップ手順 | external-integration.md, mail-templates.md, setup-guide.md |

**注:** spec-all で既にドメイン分析を実施済みの場合、その結果を再利用してよい（再分析は不要）。不足がある場合のみ追加分析を行う。

### 1A.2 結果収集と品質検証

PM オーケストレーション共通パターン（`.claude/rules/pm-orchestration.md` で自動ロード） の「結果収集と品質検証」に従う。

本スキル固有の追加基準：
- DB分析: カラム型が DB ネイティブ型（`varchar(255)` 等）で記載されていること（`string` は不可）
- API分析: リクエストパラメータ定義のキー名とレスポンスの JSON キー名が具体的に列挙されていること

### 1A.3 サブエージェント間の整合性検証

サブエージェントの結果を統合する前に、**エージェント間のデータ矛盾を検出する**：

| チェック項目 | 検証内容 |
|------------|---------|
| カラム型の一致 | DB定義のカラム型と、API分析のレスポンスフィールド型が整合しているか |
| カラム名の一致 | リクエストパラメータ定義のキー名がDBカラム名と一致しているか |
| テーブル存在 | 各機能が参照するテーブルがDB分析結果に全て存在するか |

矛盾が見つかった場合はコードを再読して正しい値を確定させる。サブエージェントの出力を鵜呑みにしない

## 設計書ファーストモード: フェーズ 1B — 要件定義書からの情報抽出

コードが存在しないため、**要件定義書と基本設計書が唯一の情報源**になる。
不足情報はユーザーに質問して補完する。

### 1B.1 Spec概要インデックスの事前作成

サブエージェント起動前に、メインエージェントが全Specを読み、**Spec概要インデックス**を作成する：
- 各REQ-IDの概要（1-2行）、関連するテーブル名、API名、画面名、外部連携の有無をインデックス化
- このインデックスを元に、各サブエージェントに**担当範囲のSpecファイルのみ**を割り当てる

### 1B.2 サブエージェントによる要件抽出

以下のサブエージェントを**並列起動**し、既存ドキュメントから設計に必要な情報を抽出する。
各サブエージェントには**担当範囲のSpecファイルのみ**を渡す（全Specを渡さない）：

| # | 名称 | 対象ファイル | 抽出内容 |
|---|------|------------|---------|
| 1 | データモデル抽出 | **DB関連のSpecのみ**（インデックスで特定）、docs/design/db-design.md | 各Specの「データモデル」セクションからテーブル・カラム情報を統合。db-design.md のテーブル定義と突合 |
| 2 | API仕様抽出 | **API関連のSpecのみ**（インデックスで特定）、docs/api/openapi.yaml | 各Specの「APIエンドポイント」セクションからパス・メソッド・パラメータを統合。openapi.yaml（APIの Single Source of Truth）と突合 |
| 3 | ビジネスロジック抽出 | **ロジックが複雑なSpecのみ**（インデックスで特定） | 各Specの「ビジネスロジック」「受入条件」セクションから業務フロー・状態遷移・計算ロジックを抽出 |
| 4 | 画面・UI抽出 | **UI関連のSpecのみ**（インデックスで特定）、docs/design/screen-flow.md | 各Specの「画面遷移図」「受入条件」から画面構成・入力要素・操作を抽出。screen-flow.md と突合 |
| 5 | 外部連携・メール・環境抽出 | **外部連携関連のSpecのみ**（インデックスで特定）、docs/design/architecture.md、CLAUDE.md | architecture.md の外部連携・技術スタック情報。各Specのメール・通知関連記述。CLAUDE.md の技術スタック・開発コマンド |

**注:** 1つのSpecが複数のサブエージェントに割り当てられることがある（例: DBもAPIもあるSpec）。その場合、各サブエージェントは自分の担当観点のセクションのみ読む。

### 1B.3 不足情報の特定と質問

サブエージェントの抽出結果を確認し、**設計に必要だが要件定義書に記載がない情報**を特定する。

**不足しやすい情報の例:**

| カテゴリ | 不足しやすい情報 | 質問例 |
|---------|----------------|--------|
| DB設計 | カラムの型・制約の具体値 | 「ユーザー名の最大文字数は？メールアドレスのユニーク制約は必要ですか？」 |
| DB設計 | インデックス戦略 | 「検索で頻繁に使うカラムは？一覧画面のソート対象カラムは？」 |
| API設計 | ページネーション仕様 | 「一覧APIのデフォルト件数は？最大件数は？」 |
| API設計 | エラーレスポンス形式 | 「エラーレスポンスのJSON形式に希望はありますか？（デフォルト: `{error: {code, message}}`）」 |
| 画面設計 | フォーム要素の詳細 | 「管理画面のフレームワーク（MUI, Ant Design等）の希望は？」 |
| 外部連携 | 認証方式・APIキー | 「WordPress APIの認証方式は？公開APIのみ使用？」 |
| 環境構築 | 具体的なバージョン | 「Ruby/Node.js/DBのバージョン指定はありますか？」 |
| マスタデータ | 初期データの具体値 | 「スキルカテゴリの初期値一覧はありますか？」 |

**質問のルール:**
- 一度に質問しすぎない（**最大10問ずつ**、カテゴリごとにまとめる）
- 「デフォルト値の提案 + 確認」の形式にする（ユーザーの負担を最小化）
  - 例: 「ページネーションのデフォルト件数は20件、最大100件でよいですか？」
- 回答がなくても設計書を書ける場合は、一般的なベストプラクティスをデフォルトとして採用し、設計書に「※ 未確定 — デフォルト値を仮採用」と注記する
- 2ラウンド質問しても回答がない項目は、デフォルト値で確定して進める

### 1B.4 情報の統合

抽出結果 + ユーザー回答を統合し、フェーズ2で使用する情報ベースを構築する。
このステップの出力は、コード分析モードのサブエージェントレポートと**同等の情報粒度**を目指す。
