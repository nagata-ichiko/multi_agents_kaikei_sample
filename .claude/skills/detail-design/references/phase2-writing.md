# フェーズ 2: 設計書執筆（spec-writer に委任）

フェーズ1の情報収集結果をもとに、**親が章立て・埋める情報・根拠を整理し、`Agent(subagent_type: spec-writer)` に委任**する。
ファイル単位で独立しているため、**ファイル単位で並列実行**する（10本以上の設計書を並列執筆できる）。

## 2.0 委任前の準備（親が実施）

spec-writer に渡す委任プロンプトを成果物ごとに組み立てる。委任プロンプトには以下を必ず含める:

1. **生成対象ファイル**: 絶対パス（例: `docs/design/db-schema.md`）
2. **参照テンプレート**: `docs/templates/phase3/` または `docs/templates/phase2/` 配下の該当テンプレート
3. **埋める情報**: フェーズ1で収集した章立て・具体値・根拠
4. **モード固有の指示**（下表参照）
5. **index系ファイルに触れない指示**（`overview.md`, `spec-map.yml`, `mkdocs.yml`, `dependency-graph.md` は親が更新する）

**モード別の委任指示:**

| ルール | コード分析モード | 設計書ファーストモード |
|--------|----------------|-------------------|
| コード根拠 | **必須**（`ファイル名:行番号` を全記述に付与） | 不要（代わりに関連Specの REQ-ID を記載） |
| 値の出典 | フェーズ1のサブエージェント抽出結果を明記 | 要件定義書 + ユーザー回答。未確定値には「※ 未確定」と注記 |
| リクエスト/レスポンス例 | 実際のコードから生成（根拠を明記） | 要件定義書のデータモデルから設計（ベストプラクティスに基づく） |
| バリデーション | モデルの validates 定義を転記（根拠を明記） | 要件定義書の受入条件から導出 |
| 未確定マーカー | 使用しない（根拠がない記述は書かない） | 不明点は「※ 未確定」を残す。推測で埋めない |

## 2.1 委任プロンプト形式

```
あなたは spec-writer です。以下の設計書を生成してください。

## 生成対象
[絶対パス（例: docs/design/db-schema.md）]

## 参照テンプレート
[テンプレートパス（例: docs/templates/phase3/db-schema.md）]

## 埋める情報
[フェーズ1で収集した情報を章立てに沿って列挙]
- 例（DB詳細スキーマ）:
  - テーブル一覧: users, posts, comments（フェーズ1 DBスキーマ分析より）
  - users テーブル: id(bigint, PK), name(varchar(255), NOT NULL), ... [根拠: schema.prisma:12-30]
  - インデックス: idx_users_email（users.email, ユニーク）[根拠: schema.prisma:45]

## コード根拠（コード分析モードのみ）
[ファイル名:行番号 の形式でフェーズ1抽出結果の根拠を列挙]

## 関連Spec（設計書ファーストモードのみ）
[REQ-ID リスト]

## 「※ 未確定」マーカーの扱い（設計書ファーストモードのみ）
- 要件定義書・ユーザー回答から判断できない値には「※ 未確定」を残す
- 推測で値を確定しない

## 厳守事項
- index系ファイル（overview.md / spec-map.yml / mkdocs.yml / dependency-graph.md）は触らない（親が更新する）
- 指示外のファイルを生成しない
- 完了時に軽量セルフチェックを実行し、結果を報告すること（mkdocs build: OK, mermaid: N/N OK, links: N/N OK）
```

## 2.2 並列委任の実行

上記プロンプトを成果物ごとに組み立て、`Agent(subagent_type: spec-writer)` を**並列起動**する。

**並列起動の単位:**
- 横断設計書は各ファイルを独立して並列実行（db-schema.md, openapi.yaml, master-data.md 等）
- 機能別設計書は 1エージェント = 最大3機能（画面・UI + ロジックの2ファイルセット）を担当
  - 例: 10機能あれば 4エージェント並列（3機能+3機能+3機能+1機能）

**並列実行できない制約:**
- `openapi.yaml` は複数エージェントが同時編集するとコンフリクトするため、1エージェントが一括担当する
- 既存ファイルの**更新**（追記）が必要な場合も1エージェントに直列でまとめる

## 2.3 結果収集

全 spec-writer 完了後、親が以下を確認する:
- 生成/更新されたファイル一覧
- 「※ 未確定」マーカーの残存箇所（ユーザーへの追加確認が必要か判断する）
- 軽量セルフチェック結果（FAIL があればフェーズ3の integrity-checker で正式チェック前に修正）

各設計書のフォーマット・ルール・コミットメッセージは [詳細設計フォーマット定義](../formats.md) を参照。成果物一覧テーブル（SKILL.md）の番号順に生成する。
